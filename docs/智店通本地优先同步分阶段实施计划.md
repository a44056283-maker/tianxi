# 智店通本地优先同步分阶段实施计划

更新时间：2026-05-12

## 1. 目标

- 自建零售系统作为业务事实源。
- 智店通作为同步目标、对账对象、外部运营系统。
- 长期依赖本地订单、库存流水、SN 台账、员工权限和同步任务，不依赖页面点击是否即时成功。

## 2. 核心对象

- `staff_user`：本地员工、角色、门店归属、启停状态，必要时映射智店通员工 ID。
- `sales_order` / `sales_order_item`：本地下单事实、行项目、支付与履约状态、SN 绑定。
- `inventory_movement`：采购入库、销售出库、退货、调拨、盘点调整。
- `serial_ledger`：SN 当前状态、最近流水、最近销售订单、成本与保修信息。
- `sync_task`：面向智店通的幂等同步任务、重试状态、错误信息、执行优先级。

## 3. 分阶段 rollout

### Phase 0: 模型定版与本地落库

- 固化上述 6 个核心实体的字段、状态流转、唯一键与幂等键。
- 本地先能独立落 `sales_order`、`inventory_movement`、`serial_ledger`，不等待智店通成功。
- 将现有 `latest-inventory-movements.json`、`latest-serial-overrides.json` 视为过渡输入，不作为最终主存储。

验收：

- 本地可独立重建某日库存与 SN 状态。
- 任一销售单可追溯到操作员工、行项目、出库流水、SN。

### Phase 1: 本地事实 -> 智店通单向同步

- 新增 `sync_task` 队列，由本地业务事件派生待同步任务。
- 先打通 `staff_user`、`sales_order`、`sales_outbound inventory_movement` 三类同步。
- 所有推送都要求幂等键、远端回执、失败重试和人工死信队列。

验收：

- 同一订单重复投递不会在智店通形成重复单。
- 智店通临时失败不影响本地下单与库存记账。

### Phase 2: SN / 库存对账与异常补偿

- 用 `serial_ledger` 和 `inventory_movement` 对账智店通销售出库、采购入库、其他出库。
- 建立日级对账任务：订单差异、SN 差异、库存数量差异。
- 差异只生成异常和补偿建议，不允许智店通直接覆盖本地事实。

验收：

- 能区分“本地未推送”“远端成功但本地未回写”“远端人工改单”三类问题。
- 每日差异具备可复核明细和重放入口。

### Phase 3: 扩展外围模块

- 再接财务、设备、报表、绩效等外围能力。
- 保持本地主链路仍是员工 -> 订单 -> 库存流水 -> SN 台账 -> 同步任务。
- UI 自动化仅保留为应急兜底，不作为主业务实现。

## 4. 推荐技术切面

- 存储：SQLite/Postgres 均可，先落结构化表，再保留现有 JSON 快照作为派生产物。
- 事件触发：本地下单、入库、调拨、退货完成后立即生成 `sync_task`。
- Worker：独立同步 worker，按任务类型串行或分组执行。
- 监控：记录任务耗时、重试次数、最近错误、远端回执 ID、死信数量。

## 5. 与当前仓库的衔接

- 继续复用已有能力：库存同步、销售出库导入、采购入库导入、SN 覆盖、零售快照。
- 新增本地业务模型时，先作为 `apps/inventory-sync/src/localRetailSync/` 的类型与任务边界。
- 后续再把 FastAPI 或独立服务接到这些边界上，而不是直接把现有导入脚本扩成全量业务系统。

## 6. 当前剩余空白

- 还没有正式数据库表和迁移。
- 还没有本地下单 API / 后台服务。
- 还没有智店通“创建订单 / 创建员工 / 出库确认”稳定接口清单。
- 还没有死信处理台、人工复核台和系统级对账报表。
