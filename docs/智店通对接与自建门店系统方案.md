# 智店通对接与自建门店系统方案

更新时间：2026-05-12

## 1. 结论

可以做，但要分层做，不能把“浏览器代点智店通页面”当成长期正式方案。

对当前项目，最稳妥的目标应该是：

1. 自建一套门店端 / 手机收银端，作为日常业务录入前台。
2. 本地服务器成为我方业务中台，先落本地订单、库存流水、SN 台账、操作员日志。
3. 智店通作为外部业务系统，先做同步和对账，再逐步推进到半自动回写。
4. 定时任务负责巡检、补同步、异常告警、陈旧库存提醒。

不建议一开始就把核心业务直接绑死在网页自动点击上。原因很简单：页面结构、权限、登录态、验证码、风控、单标签限制，都会让生产链路不稳定。

## 3. 当前项目中已具备的对接基础

代码里已经存在的基础能力：

- 登录地址配置：`https://retail-pos.lenovo.com/web/login`
- API 基地址配置：`https://retail-pos.lenovo.com/apis`
- 已有库存模块路径常量
- 已有库存快照导入和 SN 台账模型
- 已有销售出库 / 采购入库 / 其他出库 的本地同步链路
- 已有定时巡检能力

现有实现更接近“同步与校准器”，还不是“正式业务中台”。

## 4. 是否能实现与智店通对接

可以，分三层理解：

### 4.1 第一层：数据同步级

能力：

- 从智店通取库存、入库、出库、订单结果
- 同步到本地系统
- 修正网站库存、SN、陈旧库存提醒

这一层已经部分实现。

适合用途：

- 先把库存真实度做起来
- 先建立网站展示、库存追踪、价保提醒

### 4.2 第二层：业务回写级

能力：

- 我方手机端 / 收银端生成订单
- 本地服务器落单
- 同步器将订单回写到智店通
- 回写成功后，智店通形成正式订单或正式出库

这是下一步最关键的一层。

难点：

- 需要确认智店通内部接口是否允许稳定创建订单或出库单
- 需要处理登录态、权限、字段校验、异常补偿
- 需要防止重复回写

### 4.3 第三层：实时双向协同级

能力：

- 我方系统和智店通双向实时同步
- 门店端、手机端、网站端、智店通库存完全联动
- 自动对账、自动补偿、冲突处理

可以做，但这是完整中台项目，不是简单脚本增强。

## 5. 推荐的目标架构

推荐架构：

1. 门店前台层
   - 手机端
   - 收银端
   - 店长后台

2. 本地业务服务层
   - 订单服务
   - 库存服务
   - SN 台账服务
   - 员工与权限服务
   - 价格与报价服务
   - 陈旧库存与价保提醒服务

3. 同步编排层
   - 智店通入库同步器
   - 智店通出库同步器
   - 智店通订单同步器
   - 京东 / 联想官网采集器
   - 定时任务调度器

4. 对账与监控层
   - 同步日志
   - 失败重试
   - 差异审计
   - 人工复核队列

### 5.1 核心原则

- 本地服务器先记“真实业务事实”
- 智店通同步器负责把事实投递到外部系统
- 不允许前台直接依赖浏览器页面是否成功点击
- 所有入库、出库、调库、退货都必须有唯一业务单号和幂等键

## 7. 推荐的同步方向

### 7.1 我方系统 -> 智店通

适用于：

- 门店收银销售
- 非零售出库
- 商品入库
- 库位调整

流程建议：

1. 员工在我方门店端完成业务
2. 本地服务器先落订单和库存流水
3. 生成待同步任务
4. 同步器把任务投递到智店通
5. 成功后记录智店通业务单号
6. 失败则进入重试或人工复核

### 7.2 智店通 -> 我方系统

适用于：

- 厂家侧入库
- 其他人员在智店通后台手工操作的库存业务
- 已存在历史库存、历史订单

流程建议：

1. 定时扫描当天或指定时间段业务单
2. 拉取增量详情
3. 合并到本地台账
4. 标记来源为 `zhidiantong`
5. 做差异校验

## 8. 为什么不建议把网页自动化当正式主链路

原因不是“不能点”，而是“点了不稳”：

- 单浏览器单会话限制明显
- 真实账号有权限和风控限制
- 页面表格和按钮文案会改
- 弹窗、验证码、超时、加载圈都可能打断流程
- 生产故障后很难做幂等补偿

因此：

- 浏览器自动化适合审计、补录、探索流程
- 正式同步链路应尽量使用可复现的数据接口或标准化导入/导出

## 9. 对当前项目的实施阶段建议

### 阶段 1：把本地系统做成正式业务台账

目标：

- 门店端能录销售
- 手机端能录出库
- 本地库存、SN、操作员都完整留痕

必须完成：

- 员工登录
- 角色权限
- 订单创建
- SN 选择
- 库存流水生成
- 本地库存实时扣减

### 阶段 2：把智店通同步器做成单独服务

目标：

- 自建系统成为操作入口
- 智店通成为外部同步目标

必须完成：

- 出库同步任务表
- 入库同步任务表
- 同步结果回写
- 失败重试
- 差异告警

### 阶段 3：接入定时任务和提醒

目标：

- 自动巡检库存差异
- 自动提醒陈旧库存
- 自动生成待处理清单

至少包括：

- 每日销售同步
- 每日入库同步
- 陈旧库存 180 天巡检
- 智店通同步失败告警

### 阶段 4：再考虑深入对接

内容：

- 订单回写
- 退单同步
- 财务状态映射
- POS 终端对接
- 实时报表

## 10. 对你这个项目的直接判断

可以做，而且方向是对的。

但正确做法不是“继续把网页脚本写复杂”，而是：

- 把我们自己的智慧零售系统做成业务主系统
- 把智店通做成需要同步、对账、补偿的外部系统

这样后续无论是手机端、收银端、老板看板、库存零售专区，还是 SN 价保、陈旧库存提醒，都能统一建立在本地服务器之上。

## 11. 当前最值得马上落地的三件事

1. 设计本地 `订单 / 流水 / SN / 员工 / 同步任务` 数据表
2. 新增“门店收银出库”本地 API 和页面
3. 把智店通同步器升级为独立任务队列，而不是零散脚本

这三件事完成后，才算真正进入“自建门店智能化管理系统”阶段。
