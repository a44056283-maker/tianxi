# 库存及报价实时管理模块实施计划

## 1. 模块目标

本模块作为联想智慧零售系统首个优先开发模块，目标不是简单展示库存和报价，而是建立一套可以支撑门店经营决策的库存、成本、报价、保修和国补价格中台。

核心目标：

- 对接联想智店通智慧零售系统网页版，沉淀门店真实库存。
- 以序列号 SN 管理单机库存周转、入库时间、成本来源、保修状态和销售状态。
- 汇总联想订货平台、分销商日报价、灰渠公众号、电商平台价格，形成可追溯报价库。
- 自动计算两套国补价格：正规厂家渠道国补价、防流失低价国补价。
- 为销售提供可用报价、最低保护价、审批提示和服务差异提醒。
- 为老板提供库存积压、补货、清仓、毛利和价格风险看板。

## 2. 业务边界

### 2.1 库存来源

真实库存系统为联想厂家对接的联想智店通智慧零售系统：

```text
https://retail-pos.lenovo.com/web/login
```

当前厂家未提供 API Key，仅有网页版。因此一期采用浏览器自动化/RPA 同步数据，后续如果出现官方接口再替换采集层。

### 2.2 报价来源

进货报价：

- 联想订货平台：实时程度较慢，作为厂家正规渠道进货参考价。
- 分销商固定群日报价：通常每日 12 点前发送，作为正常提货价。
- 灰渠公众号报价：每日不定时发布，不含增值税，作为低价采购参考。

销售价格：

- 联想官网实时报价。
- 京东实时报价与京东国补后价格。
- 慢慢买只作为辅助定位真实详情页和历史比价线索，不直接作为门店报价来源。

采集顺序：

1. 先进入联想商城首页 `https://shop.lenovo.com.cn/`，通过首页搜索框、商城分类、国家补贴/教育特惠等入口模拟人工点击采集；不得直接把 `s.lenovo.com.cn/search` 当成采集起点。
2. 京东先进入联想自营店 `https://lenovo1.jd.com/`，自营缺货/缺型号时补充已确认备用店铺；`https://mall.jd.com/index-1000000157.html?from=pc` 不再作为采集入口。
3. 商品网址锁定库只接受详情页：京东必须是 `https://item.jd.com/数字.html`，联想必须是 `https://item.lenovo.com.cn/product/数字.html`。店铺页、商城首页、搜索页只能作为采集起点，不能写入“已锁定商品网址”。
4. 官方入口找不到详情页时，用慢慢买作为兜底线索找对应京东/联想详情页；慢慢买价格不直接进入正式报价，只有反查到官方详情页且配置完全匹配后才可写价格。
3. 先按品牌/商品分类页浏览当期在售商品。
4. 在分类页上下滚动定位同型号、同 CPU、同内存、同硬盘、同显卡配置，优先使用分类页定位到的当期在售详情页。
5. 分类页找不到时，再用 PN-MTM、主型号、CPU/GPU/内存/硬盘关键词搜索。
6. 慢慢买仅用于补充“可能的官方详情页/京东详情页线索”，最终写入报价必须回到京东自营、京东官方授权店、京东超市联想自营或联想商城真实详情页。
7. 已下架、待发布、待公布、暂不销售、无货、售罄、到货通知页面不作为报价参考，只能作为“不可销售证据”记录。
8. 一旦 SKU 与真实商品详情页匹配成功，写入 `latest-product-url-locks.json`，下次刷新直接打开锁定网址采集动态价格，不再重复全网搜索。

### 2.3 国补报价

系统需要同时维护两套国补价：

- 正规厂家渠道国补价：基于正规进货渠道、完整服务、发票和售后保障。
- 防流失低价国补价：参考京东国补价、淘宝百亿补贴价和灰渠综合成本，仅用于覆盖少量强比价客户，并需要权限与风险提示。

## 3. 数据模型规划

### 3.1 商品主数据

字段：

- 商品 ID
- 联想标准型号
- MTM / 机器型号
- 商品名称
- 品类
- 配置
- CPU
- GPU
- 内存
- 硬盘
- 屏幕
- 颜色
- 是否国补机型
- 是否主推
- 是否允许灰渠覆盖

### 3.2 单机库存 SN 数据

字段：

- SN 序列号
- 商品 ID
- 入库日期
- 入库来源：厂家、分销商、灰渠、调拨
- 入库成本
- 当前所在位置：门店、仓库、在途
- 库存天数
- 库存状态：正常、紧俏、低库存、积压、异常
- 保修开始时间
- 保修结束时间
- 服务权益
- 保修校验状态
- 是否已售
- 售出日期
- 成交价
- 毛利

### 3.3 报价来源数据

字段：

- 商品 ID
- 报价来源：厂家订货平台、分销商、灰渠公众号、京东、联想官网、淘宝百亿补贴
- 来源类型：进货价、销售参考价、国补参考价
- 价格
- 是否含税
- 是否含服务
- 发布时间
- 采集时间
- 证据链接或截图
- 可信等级

### 3.4 国补规则数据

字段：

- 城市/区域
- 国补比例
- 单台补贴上限
- 参与品类
- 正规渠道适用规则
- 低价防流失规则
- 是否需要店长审批
- 是否需要保修/服务风险提醒

### 3.5 报价决策数据

字段：

- 商品 ID
- 正规成本基准
- 灰渠成本基准
- 当前市场参考价
- 正规国补价
- 防流失国补价
- 推荐销售价
- 最低保护价
- 预计毛利
- 审批状态
- 销售话术提示
- 风险提示

## 4. 系统实现阶段

### 阶段 1：基础数据结构与前端业务原型

目标：

- 建立商品、SN 库存、报价来源、国补规则、报价决策 mock 数据。
- 将现有库存及报价面板改为贴近真实业务的首版看板。
- 前端明确展示智店通同步、SN 周转、保修校验、多来源报价、两套国补价。

交付：

- 可运行前端页面。
- 可读 mock 数据结构。
- 业务字段为后续后端/API 对接预留。

### 阶段 2：后端占位与本地数据服务

目标：

- 建立后端服务占位。
- 提供 `/api/inventory-quote/summary`、`/api/inventory-quote/serials`、`/api/inventory-quote/prices` 等接口。
- 前端从 API 读取数据，不再直接依赖 mock。

交付：

- 本地后端服务。
- 前后端接口契约。
- JSON 样例。

### 阶段 3：智店通网页同步

目标：

- 使用浏览器自动化登录智店通。
- 支持人工触发和定时同步。
- 优先同步库存、SN、入库时间、门店/仓库位置、销售出库记录。

交付：

- 智店通同步任务。
- 同步日志。
- 失败重试与人工处理入口。

### 阶段 4：SN 官网保修校验

目标：

- 自动根据 SN 查询联想官网保修起止时间与服务权益。
- 识别提前激活、保修异常、服务权益差异。
- 把保修状态写入单机库存数据。

交付：

- SN 校验队列。
- 保修异常列表。
- 销售报价时的服务风险提示。

### 阶段 5：报价采集与导入

目标：

- 支持分销商群日报价的 Excel、图片、文字半自动导入。
- 采集联想订货平台价格。
- 采集灰渠公众号报价。
- 采集京东、联想官网、淘宝百亿关键型号价格。

交付：

- 报价来源库。
- 报价可信度和更新时间。
- 价格证据链接/截图字段。

### 阶段 6：国补价格引擎

目标：

- 计算正规厂家渠道国补价。
- 计算防流失低价国补价。
- 按权限控制低价策略展示和使用。
- 输出推荐报价、最低保护价、毛利和审批提示。

交付：

- 国补价格计算规则。
- 报价助手首版。
- 店长审批规则。

### 阶段 7：经营分析与自动建议

目标：

- 库存周转、积压、补货、清仓建议。
- 毛利风险、比价压力、低价销售预警。
- 适合投放广告机、主推、清仓的 SKU 自动推荐。

交付：

- 老板经营看板。
- 销售执行建议。
- 每日库存与报价日报。

## 5. 智能化方向补充

参考开源智慧零售和库存系统后，本项目后续应向以下方向增强：

- 库存底座参考 InvenTree：采用低层级库存追踪、REST API、插件扩展、外部系统集成的设计思路。
- POS/门店能力参考 TailPOS、QPOS 等项目：保留离线可用、库存预警、动态定价、报表和权限体系。
- 报价能力独立成价格引擎：不要把价格计算写死在页面里，应支持规则版本、审批记录、价格证据、不同渠道成本基准。
- 采集任务独立成任务队列：智店通、订货平台、电商价、公众号价都应有任务状态、失败原因、重试、人工复核。
- 智能建议应基于可解释规则先落地：例如库存天数、比价压力、SN 保修状态、毛利保护线、国补上限，再逐步加入 AI 总结和异常检测。
- 报价助手应输出销售可执行结论：推荐报价、最低保护价、正规国补价、防流失价、审批线、服务风险、建议话术。

参考来源：

- InvenTree: https://github.com/inventree/InvenTree
- TailPOS: https://github.com/bailabs/tailpos
- QPOS: https://github.com/qtecsolution/qpos

## 6. 电子价签对接规划

本系统后续需要对接电子价签，建议将电子价签作为“价格执行终端”，与库存报价模块打通。

### 6.1 业务流程

```text
报价引擎生成门店执行价
  ↓
店长确认或系统自动批准
  ↓
生成电子价签更新任务
  ↓
价签平台渲染模板图片
  ↓
基站下发到对应电子价签
  ↓
回传更新状态、失败原因、电量、在线状态
```

### 6.2 系统字段

需要新增：

- 价签 ID
- 价签型号
- 绑定 SKU
- 绑定门店货架位置
- 基站 ID
- 当前显示价格
- 待更新价格
- 模板版本
- 电量
- 在线状态
- 最后下发时间
- 下发结果

### 6.3 汉朔电子价签判断

公开资料显示，汉朔 Nebular 等价签使用 2.4GHz 无线通信，系统包含 ESLs、ESL Controller、ESL-Working、PriSmart/ShopWeb、Database、Integration server、CloudMonitor 和 PDA。汉朔披露文件中也说明，无线通信基站与价签通过自研 HiLPC 协议连接，ESL-Working 内嵌私有协议，价签、基站、ESL-Working 之间相互绑定且不兼容其他品牌设备。

因此，如果已有两个原版汉朔发射器，是否能继续使用取决于：

- 发射器/基站具体型号。
- 价签型号是否与基站同系列、同频段、同区域版本。
- 是否有 ESL-Working 或 PriSmart/ShopWeb 软件授权。
- 是否能进入基站管理后台。
- 是否能拿到汉朔本地部署包、SaaS 账号或集成接口。
- 基站是否被原项目绑定、锁定或需要厂商解绑。

初步判断：

- 如果两个发射器是可用的汉朔基站，并能获得汉朔软件或接口授权，则大概率可以复用，只需要购买兼容价签。
- 如果只有基站硬件，没有 ESL-Working/PriSmart 或接口权限，不能稳定、合法地直接控制价签。
- 不建议走破解私有协议路线，后期维护、稳定性和合规风险都高。

参考来源：

- Hanshow Nebular: https://www.hanshow.com/nebular
- Hanshow Nebular Product Manual: https://fcc.report/FCC-ID/2AYMH-NEBULAR-350B/5851426.pdf
- 汉朔公开披露文件: https://reportdocs.static.szse.cn/UpFiles/rasinfodisc1/202311/RAS_202311_BBF9576B8C314289AB174EEC2F01C8F2.pdf

### 6.4 需要用户提供的信息

- 两个发射器/基站的正反面照片。
- 型号、SN、MAC、频段、供电方式。
- 是否有原配电源、网线、安装支架。
- 是否知道原来的管理平台、账号或部署服务商。
- 准备购买的价签型号、尺寸、颜色能力。
- 门店面积、货架数量、预计价签数量。
- 是否希望价签显示二维码、国补价、活动价、库存状态或服务权益。

## 7. 需要业务方提供的信息

### 7.1 智店通

- 登录方式说明，不直接在文档或聊天中保存密码。
- 是否有短信、滑块、扫码、人脸等验证。
- 库存页、入库页、销售页、序列号页截图。
- 是否能导出 Excel。
- 脱敏 Excel 样例。
- 门店与仓库数量。
- 是否能看到 SN、入库价、入库时间、历史销售。

### 7.2 联想订货平台

- 平台网址。
- 登录方式。
- 报价页面截图。
- 是否支持导出。
- 报价字段样例。
- 是否区分区域、客户等级和活动资源。

### 7.3 分销商日报价

- 群平台：微信、企业微信、QQ、飞书、钉钉等。
- 最近 7 天报价样例。
- 报价格式：文字、图片、Excel、PDF。
- 是否含税、物流、账期、返利。
- 是否有多个分销商报价需要比较。

### 7.4 灰渠公众号

- 公众号名称。
- 最近 5-10 篇报价样例。
- 报价形式：文章、图片、表格、小程序或链接。
- 是否全部不含税。
- 是否存在区域、数量、颜色限制。

### 7.5 电商平台

- 需要重点采集的 SKU 清单。
- 京东是否只采自营/官方旗舰店。
- 淘宝百亿是否只采热销型号。
- 是否要采集优惠券、满减、国补后价。
- 是否需要保存价格截图作为证据。

### 7.6 国补和报价规则

- 当前适用城市/区域。
- 国补比例和单台上限。
- 参与品类和限制条件。
- 正规渠道国补操作规则。
- 京东/淘宝国补价参考方式。
- 各品类目标毛利。
- 最低保护毛利。
- 普通销售让利额度。
- 店长审批线。
- 哪些型号禁止低价卖。
- 哪些型号允许灰渠覆盖。

## 8. 当前开发原则

- 先做业务闭环，再做自动化采集。
- 先支持人工导入和模拟数据，再替换为网页采集。
- 所有外部来源价格都保留来源、时间、证据和可信度。
- 正规渠道货和灰渠货在数据层必须区分。
- 低价防流失报价必须有权限、审批和风险提示。
- SN 保修状态必须成为报价与交付的重要依据。
