# 电脑操控实机能力审计（2026-05-15）

更新时间：2026-05-15

## 目标

记录当前这台 Mac 上，联想智慧零售项目实际可用的电脑操控能力，而不是只停留在“理论上有技能”。

## 本轮实机验证范围

1. `computer-use` 是否能真实读取关键 App
2. 哪个浏览器更适合作为远程看板和业务页面承载
3. 微信当前在 `computer-use` 下的真实限制
4. OCR 增强层当前本机就绪度

## 已验证 App

### 1. 微信

App 路径：

- `/Applications/WeChat 2.app/`

当前结果：

- `list_apps` 可识别
- `get_app_state` 可连接到窗口
- 但当前可访问性树只读到：
  - 窗口框架
  - 菜单栏
- 当前截图表现为整窗空白

当前判定：

- 微信已被 `computer-use` 识别
- 但当前 **不适合依赖深层 Accessibility 控件树做稳定自动点击**
- 当前更适合：
  - 用户先手动打开目标会话/公众号
  - 再做可见界面辅助读取

### 2. Google Chrome

App 路径：

- `/Applications/Google Chrome.app/`

当前结果：

- `get_app_state` 可读取：
  - 地址栏
  - 标签页
  - 页面主内容
  - 前端按钮、筛选器、链接和文本
- 已验证可直接读取：
  - `127.0.0.1:5174`
  - 联想智慧零售系统主页面结构

当前判定：

- Chrome 当前是 **业务网页核验主力浏览器**
- 适合：
  - 京东
  - 联想官网
  - 智店通
  - 本地 `5174`

### 3. 360 极速浏览器

App 路径：

- `/Applications/360极速浏览器.app/`

当前结果：

- `get_app_state` 可读取窗口和地址栏
- 已验证当前可见页面：
  - `http://192.168.13.104:5174/`
- 当前页面截图和窗口结构可正常返回

当前判定：

- 360 浏览器当前也可作为：
  - 局域网远程页面验证
  - 备用网页登录容器
- 但项目主规则仍优先以 Chrome 稳定会话为主

## OCR 增强层审计

### 当前已存在

系统：

- `/usr/sbin/screencapture`

Python：

- `PIL`
- `numpy`

### 当前缺失

系统二进制：

- `tesseract`
- `ocrmypdf`

Python 包：

- `pytesseract`
- `easyocr`
- `cv2`
- `rapidocr_onnxruntime`

Node 包：

- `tesseract.js`
- `sharp`
- `playwright`

### 当前判定

1. 当前机器上还没有可直接拿来用的成熟 OCR 主链路
2. 现阶段只能先依赖：
   - 可见页面
   - 手工读取
   - 原文 manual 文件落地
3. 下一步如果要增强，最优先不是微信桥接，而是先补 OCR 层

## 当前正式结论

1. 微信：
   - 已识别
   - 但可访问性树过浅，当前不适合作为稳定深控对象
2. Chrome：
   - 当前最稳定
   - 适合业务网页核验
3. 360：
   - 可作为远程页面验证备用容器
4. OCR：
   - 本机当前未成型
   - 要继续增强时，优先补 OCR，不优先补微信桥接

## 本轮新增的项目内承接能力

虽然 OCR 引擎还没装，本轮已经把“微信截图证据 -> 系统登记 -> 待 OCR 队列”接进项目：

1. 证据索引：
   - `apps/inventory-sync/artifacts/latest-wechat-quote-evidence.json`
   - `apps/web-cockpit/public/data/latest-wechat-quote-evidence.json`
2. OCR 待处理队列：
   - `apps/inventory-sync/artifacts/latest-quote-ocr-queue.json`
   - `apps/web-cockpit/public/data/latest-quote-ocr-queue.json`
3. CLI 命令：
   - `node --import tsx/esm src/cli.ts build-wechat-quote-evidence`
   - `node --import tsx/esm src/cli.ts build-quote-ocr-queue`

当前实测结果：

- 已识别到 `2` 张灰渠公众号截图证据
- 当前 `pending_ocr = 2`
- 当前还没有同日期 `gray-wholesale-YYYY-MM-DD.txt/.md`
- 当前 OCR 运行层已可直接使用：
  - `python3 scripts/ocr_wechat_quote_image.py <image> --text-out <txt> --json-out <json>`

这意味着后续即使还没安装 OCR，系统也已经能明确知道：

1. 哪些公众号截图已经存在
2. 哪些截图还没转成文本
3. 哪些日期缺 manual 原文

## 对项目执行顺序的影响

当前正式顺序继续保持：

1. 微信任务：
   - `chronicle -> computer-use -> manual 文件 -> 本地解析`
2. 网页任务：
   - `chronicle -> chrome -> 必要时 computer-use`
3. 增强顺序：
   - 先 OCR
   - 再 macOS 控制增强
   - 最后才评估微信桥接
