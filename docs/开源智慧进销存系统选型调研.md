# 开源智慧进销存系统选型调研

更新时间：2026-05-12

## 1. 选型目标

当前项目已经有：

- 联想门店智慧零售驾驶舱
- 智店通库存、SN、销售出库、采购入库同步原型
- 本地销售单、库存流水、同步任务 API 骨架
- 后续电子价签对接需求

本次调研目标不是直接替换现有项目，而是找到可借鉴、可整合、可拆模块吸收的开源系统，为后续形成“智慧零售进销存软件”提供参考代码和架构依据。

## 2. 优先推荐组合

### A. InvenTree：库存/SN/接口底座参考

地址：https://github.com/inventree/InvenTree

定位：

- 开源库存管理系统
- 强项是低层级库存追踪、部件/库存/位置/批次管理
- 提供 REST API、Python API、插件系统
- 后端为 Python/Django/DRF，支持 PostgreSQL/MySQL/SQLite
- 许可证：MIT

适合我们借鉴：

- SN 单机库存模型
- 库存位置、库存流水、库存状态设计
- API-first 和插件式外部系统集成方式
- 后续对接智店通、电子价签、保修查询时的插件边界

不适合直接照搬：

- 它偏工程/零部件库存，不是门店 POS
- 没有完整门店收银、销售话术、国补报价、智店通对账这些本项目特有能力

结论：

- 作为库存与 SN 台账的主要参考对象。
- 不建议直接把它作为整套系统底座迁入，但建议吸收它的数据模型和 API 设计。

### B. ERPNext：完整进销存/ERP/POS 参考

地址：https://github.com/frappe/erpnext

定位：

- 开源 ERP
- 覆盖采购、销售、库存、订单、会计、CRM、POS 等完整业务
- 基于 Frappe/Python，提供 REST API
- 适合研究完整进销存业务闭环

适合我们借鉴：

- 销售订单、采购订单、库存凭证、仓库、客户、供应商模型
- POS 与库存扣减之间的关系
- 进销存报表口径
- 权限、角色、流程审批

不适合直接照搬：

- 系统重，业务范围远大于单店联想体验店
- 与当前 FastAPI + React 项目技术栈不完全一致
- 二次开发成本和部署复杂度较高

结论：

- 作为进销存业务闭环和报表口径参考。
- 我们继续保持自建轻量系统，按需吸收 ERPNext 的业务对象。

### C. Open Source Point of Sale：门店 POS 流程参考

地址：https://github.com/opensourcepos/opensourcepos

定位：

- Web POS 系统
- 包含商品、库存、销售、收货、客户、供应商、权限、报表
- 技术栈 PHP/CodeIgniter/MySQL

适合我们借鉴：

- 收银台销售流程
- 退货、收货、库存调整、权限、报表
- 线下门店小团队使用方式

不适合直接照搬：

- 技术栈与当前项目差异较大
- 界面和架构不适合作为我们智慧驾驶舱的直接底座

结论：

- 作为门店端“收银出库页面”和基础 POS 流程参考。

### D. SaleFlex / RetailPOS / Olgax POS：离线 POS 与多端体验参考

候选：

- SaleFlex.PyPOS：https://github.com/SaleFlex/SaleFlex.PyPOS
- RetailPOS：https://retailpos.org/
- Olgax POS：https://github.com/OLGAX-com/olgax-pos

适合我们借鉴：

- 离线优先 POS
- 本地交易队列
- 断网后同步
- 收银触屏体验
- 小店自托管部署方式

风险：

- SaleFlex 当前仍处于 beta/早期阶段，仓库活跃度和成熟度需要进一步验证。
- RetailPOS 更偏电商平台 POS 同步。
- Olgax POS 仍是 MVP，适合参考实现方式，不适合作为主底座。

结论：

- 用于参考“本地销售单 -> 同步队列 -> 外部平台回写”的模式。
- 当前不建议直接迁入。

### E. OpenEPaperLink：电子价签开源对接参考

地址：https://github.com/OpenEPaperLink/OpenEPaperLink

定位：

- 电子纸价签开源固件与协议实现
- 支持 ESP32 接入点和多种电子纸标签
- 可生成并下发价签显示内容
- 非商业许可证：CC BY-NC-SA 4.0

适合我们借鉴：

- 电子价签网关、标签、模板、下发队列的系统边界
- 价签图片生成与任务下发流程
- 标签在线状态、电量、信号、刷新状态模型

关键限制：

- 许可证不适合直接用于商业门店软件分发。
- 真实商用电子价签通常来自汉朔、SOLUM、SES-imagotag 等厂商，最终仍要对接厂商 API 或中间服务器。

结论：

- 作为电子价签模块的数据模型和技术验证参考。
- 商业落地时，应把电子价签做成独立适配层：`price_tag_device`、`price_tag_binding`、`price_tag_update_task`、`price_tag_template`。

### F. Odoo + SES-imagotag 模块：电子价签业务集成参考

地址：https://github.com/bascouba/mymodules

定位：

- Odoo 的 SES-imagotag 电子价签管理模块
- 支持价签、模板、商品绑定、刷新价签

适合我们借鉴：

- ERP/POS 商品与电子价签绑定关系
- 模板管理
- 价签刷新任务

风险：

- 仓库较老，不能直接作为生产依赖
- 绑定的是特定厂商生态

结论：

- 适合作为“电子价签业务对象”的参考，不作为代码依赖。

## 3. 建议整合架构

建议保留当前项目为主系统，不把大 ERP 直接搬进来。

推荐路线：

1. 库存底座参考 InvenTree，建立本地标准表：商品、SKU、SN、仓库/库位、库存流水、库存状态。
2. 进销存闭环参考 ERPNext，建立销售单、采购单、供应商、客户、退货、调整、对账。
3. 门店 POS 流程参考 Open Source Point of Sale，落地“门店收银出库”和“手机扫码出库”。
4. 智店通作为外部同步目标，保留本地 `sync_task` 队列、重试、失败原因、人工补偿。
5. 电子价签作为独立适配层，先做标准任务表，再按实际厂商 API 接入。

## 5. 结论

最适合当前项目的开源参考组合是：

- 库存/SN/API：InvenTree
- 进销存闭环：ERPNext
- 门店 POS 流程：Open Source Point of Sale
- 离线销售同步：SaleFlex / RetailPOS / Olgax POS
- 电子价签技术验证：OpenEPaperLink
- 商用电子价签业务模型：Odoo SES-imagotag 模块

下一步开发不建议“大迁移”，而是按我们现有 FastAPI + React 工程继续建设，把上述项目中成熟的数据模型和流程拆解吸收进来。
