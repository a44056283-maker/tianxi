# 出入库自动同步全链路审计

时间：2026-05-31

范围：
- OpenClaw / ZDT PostgreSQL
- 本地 SQLite `retail-core.sqlite3`
- FastAPI 聚合接口
- 前端 `入库出库` 页面

目标：
- 查清为什么会出现重复、混乱、无价格、实付金额单位错误、分类错位。
- 明确根因、风险、修复顺序，避免“采完同步又恢复错误形态”。

---

## 1. 当前链路总览

当前不是单主链，而是多条链同时写同一业务面：

1. OpenClaw 全量订单链
   - `apps/api-server/app/retail_core.py`
   - `sync_all_sales_orders_from_openclaw_sql(...)`
2. ZDT 销售出库流水派生销售单链
   - `apps/api-server/app/retail_core.py`
   - `sync_sales_orders_from_inventory_movements(...)`
3. 智店通导出快照销售单链
   - `apps/api-server/app/retail_core.py`
   - `sync_sales_orders_from_export_snapshot(...)`
4. ZDT 库存流水增量链
   - `apps/inventory-sync/src/storage/zdtSqliteSync.py`
5. 前端显示链
   - `apps/api-server/app/retail_core.py`
   - `list_inventory_movements(...)`
   - `list_sales_orders(...)`
   - `apps/web-cockpit/src/App.tsx`

结论：

- 当前销售单、库存流水、前端分类不是一条链出来的，而是多条链并写，再由前端二次猜测合并。
- 这就是“修完一层，另一层又把错误刷回来”的根因。

---

## 2. 已确认的真实问题

### P0-1 销售金额单位错误

证据：

- SQLite `sales_order` 最新订单金额为：
  - `XS26053194481297419 -> pay_amount = 129900.0`
  - `XS26053111105287919 -> pay_amount = 579900.0`
- 这明显是“分”被当成“元”展示。

来源代码：

- `apps/api-server/app/retail_core.py`
  - `sync_all_sales_orders_from_openclaw_sql(...)`
  - 直接把 `fact_orders.total_amount / pay_amount` 原值写进 SQLite
- `apps/api-server/app/retail_core.py`
  - `enrich_sales_orders_from_openclaw_sql(...)`
  - 同样直接回填金额

风险：

- 前端“实付金额”直接错 100 倍。
- 客户资料、累计实付、价保金额判断都会被污染。

结论：

- `fact_orders` 这条链当前金额字段必须做单位标准化，统一落成“元”。

---

### P0-2 采购/入库单被错误写进销售单表

证据：

- SQLite `sales_order` 中存在：
  - `ZDT-CGR260531422330_1384237798599913472`
  - `ZDT-CGR260531422299_1384237798599913472`
  - `ZDT-CGR260531422229_1384237798599913472`
- 这些明显是采购入库单号，却被写成：
  - `order_type_name = 线下`
  - `total_amount = 0`

来源代码：

- `apps/inventory-sync/src/storage/zdtSqliteSync.py`
  - `ZDT_TYPE_MAP` 未覆盖或错误兜底
  - 未识别类型默认落成 `sales_outbound`
- `apps/api-server/app/retail_core.py`
  - `sync_sales_orders_from_inventory_movements(...)`
  - 只要 `movement_type = sales_outbound` 就直接派生 `sales_order`

风险：

- 采购/调拨/待商确认被塞进销售单表。
- 前端销售出库栏目必须靠启发式函数再排脏数据，必然不稳定。

结论：

- 任何非真实销售单据不得再进入 `sales_order`。
- `sales_order` 必须只由“真实销售订单事实链”生成。

---

### P0-3 同一真实订单被写出多份

证据：

- `sales_order` 逻辑重复：
  - `XS26053063268324219`
  - `ZDT-XS26053063268324219`
  - `SALE-EXPORT-XS26053063268324219-20003366-YX0JRQ9F`
- `inventory_movement` 也存在同单重复：
  - 同一 `source_ref + sku_key` 最多 3 条

来源代码：

- `sync_all_sales_orders_from_openclaw_sql(...)`
- `sync_sales_orders_from_inventory_movements(...)`
- `sync_sales_orders_from_export_snapshot(...)`
- 三条链都能写 `sales_order`
- `list_inventory_movements(...)` 又在接口层临时去重

风险：

- 数据库脏数据会反复存在。
- 只在前端或 API 层去重，换个接口、换个排序、换个重建就会复发。

结论：

- 必须把“逻辑订单唯一键”落到数据库主链，禁止三条链平行造单。

---

### P0-4 采购入库订单主链根本没落地

证据：

- SQLite：
  - `purchase_order = 0`
  - `purchase_order_line = 0`
- 但 `inventory_movement` 里已有 `purchase_inbound = 490`

风险：

- 入库只有流水，没有订单主链。
- 前端无法按真实采购单做供应商、成本、入库单号、对账闭环。

结论：

- 采购入库现在只有“流水半链”，还没有“订单主链”。

---

### P0-5 库存流水金额与销售订单金额口径分裂

证据：

- `inventory_movement.amount` 有的是 `None`
- 有的是类似 `599900.0`
- `sales_order.pay_amount` 也是 `579900.0`
- 前端在 `App.tsx` 中又会：
  - 先找 `salesOrderPrice`
  - 再从行项目 `dealPrice`
  - 找不到时拿 `totalAmount / quantity`

来源代码：

- `apps/web-cockpit/src/App.tsx`
  - `9807-9815`
- `apps/api-server/app/retail_core.py`
  - `list_sales_orders(...)`
  - `list_inventory_movements(...)`

风险：

- 金额来源不唯一。
- 同一订单不同页面显示不同金额。

结论：

- “实付金额”必须定义唯一来源，不能再让前端猜。

---

### P0-6 前端分类靠启发式猜，不是靠数据库业务分类

证据：

- `apps/web-cockpit/src/App.tsx`
  - `1506-1568`
- 当前用：
  - 单号前缀
  - note 文本
  - `sourceDocumentType`
  - “入库/采购/调拨”关键字
  来猜销售/采购/调拨/其他

风险：

- 采购单、调拨单、退单、占位单会串栏。
- 后端数据一脏，前端必错。

结论：

- 分类必须在 SQLite/API 层固化成标准业务字段，前端只消费，不再猜。

---

## 3. 根因归纳

当前问题不是一个 bug，而是四类结构性问题叠加：

1. 多写入源并行写同一业务实体
2. 金额单位未统一
3. 订单主链和流水主链未分层
4. 前端承担了本该由数据库/API 处理的去重和分类

---

## 4. 真实解决方案

### 第一层：只保留一个销售订单主链

规则：

- `sales_order / sales_order_line` 只允许来自“真实销售订单事实”：
  - OpenClaw `fact_orders + fact_order_items`
  - 或智店通导出 `orderData + orderProductData`
- `inventory_movement -> sync_sales_orders_from_inventory_movements(...)`
  只能用于补缺口登记，不允许再直接造正式 `sales_order`

必须整改：

1. 停止 `sync_sales_orders_from_inventory_movements(...)` 直接生成正式销售单
2. 它只能写：
   - `order_sync_registry`
   - `sync_gap_queue`
3. 真实订单金额未到位时：
   - 库存/SN 可同步
   - 订单状态记 `pending_amount_snapshot`
   - 绝不写 0 金额正式单

---

### 第二层：统一金额单位为“元”

规则：

- OpenClaw / PG 进入 SQLite 前统一标准化：
  - `分 -> 元`
- SQLite 内所有金额字段固定为元：
  - `sales_order.total_amount`
  - `sales_order.pay_amount`
  - `sales_order_line.deal_price`
  - `sales_order_line.pay_amount`
  - `inventory_movement.amount`
  - `inventory_movement.unit_cost`

建议：

- 新增统一函数：
  - `normalize_money_to_yuan(value, source_system, field_name)`
- 在 OpenClaw 销售同步和补全链统一调用

---

### 第三层：订单唯一键与去重下沉到数据库层

规则：

- 统一逻辑订单键：
  - `canonical_order_no`
- 归一化规则：
  - `XS...` 保留原单号
  - `ZDT-XS...` -> 归一到 `XS...`
  - `SALE-EXPORT-XS...-sku-sn` -> 归一到 `XS...`
- 销售订单唯一约束应该基于：
  - `canonical_order_no`

库存流水去重键：

- 真实串号行：
  - `canonical_order_no + sku_key + serial_number`
- 无串号汇总行：
  - `canonical_order_no + sku_key + business_date + source_system`

要求：

- 不能再只在 `list_inventory_movements()` 里临时 `_dedupe_inventory_movement_rows(...)`
- 必须在同步入 SQLite 时完成去重和冲突归档

---

### 第四层：采购入库补齐正式订单主链

要求：

- 由 OpenClaw / ZDT SQL 补正式：
  - `purchase_order`
  - `purchase_order_line`
- 字段至少包括：
  - 单号
  - 供应商
  - 业务日期
  - SKU
  - 数量
  - 成本价
  - SN 列表
  - 入库操作人

否则：

- 前端采购入库永远只能看流水，无法对账。

---

### 第五层：前端取消“猜分类、猜金额”

前端应只吃 API 已固化字段：

- `business_scope`
  - `sales_order`
  - `purchase_order`
  - `transfer_outbound`
  - `transfer_inbound`
  - `other_outbound`
  - `refund_inbound`
- `amount_status`
  - `ready`
  - `pending_snapshot`
  - `not_applicable`
- `pay_amount_yuan`
- `cost_amount_yuan`
- `canonical_order_no`

前端不再通过：

- 单号前缀
- note 关键字
- sourceDocumentType 文本
做二次猜测。

---

## 5. 修复执行顺序

### P0

1. 禁止 `inventory_movement -> sales_order` 继续造正式单
2. 修 `zdtSqliteSync.py` 类型映射和默认兜底
3. OpenClaw 金额统一转元
4. 清理 `sales_order` 中非销售单、重复单
5. 清理 `inventory_movement` 中重复销售出库行

### P1

6. 补 `purchase_order / purchase_order_line`
7. API 输出 `canonical_order_no / business_scope / amount_status`
8. 前端改成只按 API 分类字段分栏

### P2

9. 对历史数据做一次性回填重建
10. 加数据库级审计任务，防止回归

---

## 6. 本轮结论

当前“重复、混乱、无价格、实付金额单位错误、分类错位”的根因已经明确，不是前端单点问题，而是：

- 同一业务实体被多条同步链重复写入
- 金额单位未统一
- 销售单和库存流水职责没切开
- 前端承担了去重与分类职责

因此真实修法不是再补提示词，而是：

1. 收敛主链
2. 统一金额口径
3. 去重下沉到数据库
4. 前端只消费标准分类字段

不把这四件事做掉，后续再采集、再同步，错误还会回刷。
