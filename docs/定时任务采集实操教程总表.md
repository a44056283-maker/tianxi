# 定时任务采集实操教程总表

更新时间：2026-05-17

## 目标

把当前会进入正式定时链的采集任务，全部收口成可训练、可验收、可重复执行的教程。后续规则固定为：

1. 先按本教程做人工演示。
2. 再让执行体按本教程做练习。
3. 练习通过后，才允许进入正式定时。
4. 到点未启动、启动未收口、页面阻塞、缺原始输入，都必须主动上报，不能静默。

## 当前固定执行口径

从现在开始，正式定时任务统一写死为：

1. 执行主体：`Codex`
2. 网页采集浏览器：默认复用用户当前已登录的 `Chrome`
3. 微信采集入口：固定为 `https://localhost:3001/`
4. 不再默认依赖 `Claude Desktop / OpenClaw / 浏览器插件 / 外接 MCP`
5. 只允许复用以下已登录地址：
   - `https://www.jd.com/`
   - `https://localhost:3001/`
   - `https://shop.lenovo.com.cn/`
   - `https://www.tmall.com/`
   - `https://retail-pos.lenovo.com/`
6. 不允许为了采集信息再打开新的未登录 Chrome 窗口、访客窗口、无痕窗口或其它浏览器

## 通用执行规则

### 所有采集统一前置规则

所有采集任务，不分微信、公众号、智店通、京东、联想官网、天猫、淘宝、桌面程序，统一必须遵守：

1. 先扫描目标当前状态
2. 判断当前是否处于可操作状态
3. 只执行一次手工单步动作
4. 动作后再次扫描状态
5. 状态正确才允许下一步

固定检查项：

- 当前窗口/标签页是不是目标
- 当前是不是已登录
- 当前是不是已加载完成
- 当前有没有白屏、转圈、验证码、403、安全验证、错误弹层、菜单栏收缩、非目标页
- 当前目标控件是不是稳定可见

固定禁止项：

- 未扫状态就点
- 连点
- 双击补救
- 盲点正文区、文章区、表格区、底部菜单区
- 在错误状态下继续硬操作

固定阻塞口径：

- 只要扫描结果不是“目标页稳定可操作”，就必须立即停下
- 状态只能写阻塞或未收口，不能假装继续推进

后续任何正式定时任务、练习任务、子代理任务单，都必须原样带上下面这段：


## 教程：竞品笔记本排行 `daily-competitor-monitor-check`

### 任务定位

- 目标：每天真实更新 THINK、华硕、惠普、华为四个品牌京东自营笔记本销量排行。
- 排行不是固定型号清单；每次都必须从店铺笔记本大类的销量排序重新形成。
- 固定详情页只用于核价、活动复核和旧商品变化比对，不能作为当天排行来源。

### 演示步骤

1. 使用用户当前已登录的 Chrome。
2. 依次打开四个品牌京东自营店铺收藏入口。
3. 进入笔记本大类；没有明确入口时，使用店内搜索 `笔记本`。
4. 点击 `销量` 排序。
5. 按当前页面顺序记录 TOP 10 商品链接与标题。
6. 逐个打开详情页核验价格、配置、国补价、活动和评价/销量文本。
7. 下架、链接失效、配置不符、掉出销量榜的旧商品，必须从当前销量排序结果替换。
8. 写入 `apps/inventory-sync/artifacts/manual/competitor-monitor-YYYY-MM-DD.json`。
9. 执行：
   - `cd apps/inventory-sync && npm run parse:competitor-monitor -- artifacts/manual/competitor-monitor-YYYY-MM-DD.json`

### 禁止项

1. 禁止用昨天固定链接顺序当今天排行。
2. 禁止用全站排行榜代替店铺自营销量排序。
3. 禁止用首页非笔记本热区代替笔记本大类。
4. 禁止用旧 JSON 重跑冒充今日采集。
5. 禁止保留下架商品作为有效排行项。

### 验收标准

1. 四个品牌都来自京东自营店铺笔记本大类销量排序。
2. 每个品牌都有当日 TOP 位次记录。
3. 前端 `报价来源 -> 竞品监控` 能看到更新后的品牌、链接、价格和活动信息。
4. 若任一品牌无法采集，任务状态必须是阻塞或未收口，不能写真实完成。

### 任务定位

- 目标：同步当天销售出库、采购入库、其他出库，并更新库存流水链。
- 原始输入：当天智店通导出或当天页面确认后的导入文件。
- 不是只读任务，是真实同步任务。
- 库存真值固定口径：
  1. 先同步当天入库、出库、零售销售、其他出库
  2. 再导出当天完整 `商品库存统计` 与 `商品库存SN统计`
  3. 最后只用这对总表统一刷新前端库存与 SN
- 库存待补信息也并入同一轮：
  - 以 `latest-inventory-master-snapshot.json` 的异常与缺口为准
  - 不能继续沿用旧的待补快照
- 禁止回到“一个商品一个商品去单采”的旧路径。

### 演示步骤

1. 先确认智店通已登录。
2. 确认不是登录页、白屏页、长时间转圈页。
3. 准备当天原始输入：
   - 销售导出
   - 采购网页导入结果
   - 其他出库导入结果
4. 在进入智店通页面前，先进入网页微信 `智店通入库群`，从上次成功扫描时间、上次阻塞时间或上次未扫描时间节点开始滚动补扫历史图片，确认教育补代扫箱码；图片被遮挡、缩略图不完整或外箱码看不清时必须打开图片或继续滚动定位。
5. 群采集必须形成图片清单并逐张确认；至少完成历史方向扫描、最新方向复扫、可疑区域二次回扫。一直扫到最后一次更新箱码代扫教育补的箱子，且清单无未确认图片，才算群采集完成。
6. 完成当天入库 / 出库导入后，立刻手动导出：
   - `商品库存统计_YYYY-MM-DD.xlsx`
   - `商品库存SN统计_YYYY-MM-DD.xlsx`
7. 只有完成网页微信和智店通可见页面手动采集证据后，才执行：
   - `bash scripts/run_scheduled_task.sh zhidiantong-sync-cycle`
   - 该命令只负责本地导入、重建、校验和报告，不允许被描述为采集器。
8. 读取报告中对应任务条目。
9. 核查：
   - 是否写入当天新记录
   - 是否确认了当天成对总表
   - 是否是 `real_completed`
   - 若不是，卡在缺原始输入、缺当天成对总表，还是页面阻塞

### 训练验收标准

1. 能准确识别智店通是否可操作。
2. 能说明当天原始输入是否齐全。
3. 能说明当天 `商品库存统计` 和 `商品库存SN统计` 是否已成对导出。
4. 能正确区分“手动可见采集证据”和“本地导入校验命令”，并读报告。
5. 不把旧文件重跑写成当天同步。

### 禁止项

1. 禁止脚本式采集、无头浏览器、新浏览器 Profile、DOM 自动扫描、高频点击冒充当天采集。
2. 禁止调用、恢复或新增旧任务名 `zhidiantong-sync-12` / `zhidiantong-sync-15` / `zhidiantong-sync-19`。
3. 没有同日网页微信 `智店通入库群` 教育补代扫证据或同日无新增确认时，不能写 `real_completed`。
4. 没有从上次未扫描时间点追溯补扫历史图片时，不能写同日无新增确认。
5. 没有图片清单、没有逐张打开可疑图片、没有二次复扫时，不能写群采集完成。

### 常见问题

1. 页面转圈超过 5 秒：
   - 退出当前页
   - 回首页稳定入口
   - 重新进入
2. 没有当天完整库存数量表和库存SN表：
   - 只能写 `executed_not_closed`
   - 不能写库存同步已完成
3. 登录失效：
   - 只能写 `blocked_page_risk`

### 同步后的前端刷新补充教程

智店通同步跑完后，库存与 SN 前端刷新链必须再参考：

- `docs/入库出库后库存与SN前端同步教程.md`

重点不是只看导入报告，而是同步核对：

1. `latest-inventory-movements.json`
2. `latest-serial-overrides.json`
3. `latest-standard-inventory-snapshot.json`
4. `latest-inventory-master-snapshot.json`
5. `latest-adjusted-inventory-snapshot.json` 只做兜底

## 教程 5：竞品监控 `daily-competitor-monitor-check`

### 任务定位

- 来源固定为 Chrome 已登录京东会话中的京东自营对应店铺或已沉淀详情页。
- 这不是“只跑定时解析”的任务，必须先形成当天真实竞品排行采集结果。
- 品牌顺序固定：
  - Think
  - 华硕
  - 惠普
  - 华为

- 禁止用京东全站排行页、无头脚本、旧 JSON、第三方店铺或非自营链接冒充当天采集。

### 演示步骤

1. 生成当天采集计划：
   - `cd apps/inventory-sync && npm run build:competitor-collection-plan`
2. 读取：
   - `apps/inventory-sync/artifacts/latest-competitor-collection-plan.json`
3. 使用 Chrome 已登录京东会话，按计划打开已沉淀详情页；链接不足的品牌打开对应京东自营店铺入口。
4. 每条候选必须点开详情页逐条核对：
   - 品牌与笔记本类目
   - 配置
   - 国补前价
   - 国补后价
   - 活动
   - 教育补贴
   - 评论量或销量文本
   - 京东自营对应店铺口径
5. 保存当天原始记录：
   - `apps/inventory-sync/artifacts/manual/competitor-monitor-YYYY-MM-DD.json`
6. 执行：
   - `bash scripts/run_scheduled_task.sh daily-competitor-monitor-check`
7. 读取任务报告和前端快照：
   - `apps/inventory-sync/artifacts/latest-scheduled-task-reports.json`
   - `apps/inventory-sync/artifacts/latest-competitor-monitor.json`
   - `apps/web-cockpit/public/data/latest-competitor-monitor.json`

### 当天 JSON 字段

每条 `items` 至少包含：

- `brand`
- `rank`
- `productName`
- `configSummary`
- `jdSelfPrice` 或 `jdPreSubsidyPrice`
- `jdSubsidyPrice`
- `jdUrl`
- `sourceStoreName`
- `storeScope = jd-self-operated-store`
- `capturedAt`
- `note`

### 成功标准

1. 当天 `competitor-monitor-YYYY-MM-DD.json` 存在。
2. 至少一个品牌有当天真实京东自营详情页记录；正常目标是四个品牌都覆盖。
3. `latest-competitor-monitor.json` 的 `quoteDate` 为当天，`isCarriedForward = false`。
4. 前端 `竞品监控` 卡片按品牌显示最新排行。

### 阻塞标准

1. 京东未登录、403、验证码、风控、安全验证。
2. 店铺页或详情页打不开。
3. 只能拿到全站排行页或第三方店铺链接。
4. 没有当天原始 JSON。

以上只能写 `blocked_page_risk` 或 `blocked_missing_input`，不能写成真实完成。

## 教程 6：SN 保修补齐 `sn-warranty-backfill`

### 任务定位

- 这是“缺口补齐”任务，不是“全量重采”任务。
- 已经采过并写入本地数据库的质保时间，视为固化结果：
  - 不再重复采
  - 不再回退成待补
  - 不允许被失败、验证码或空结果覆盖
- 本任务只处理两类 SN：
  1. 当天新入库、首次进入质保队列的 SN
  2. 前端当前仍不显示质保时间的待补 SN

### 真值来源与持久化规则

1. 官网保修查询结果是首次采集来源。
2. SQLite `serial_item` 是持久化主层。
3. `latest-lenovo-warranty-snapshot.json` 是批次快照层。
4. 一旦 `official_warranty_start / official_warranty_end` 已落地：
   - 后续任务只允许补强
   - 不允许回退为空
   - 不允许重新打成待补

### 演示步骤

1. 先确认联想保修查询页已登录、可操作。
2. 读取待补队列：
   - `apps/inventory-sync/artifacts/latest-warranty-check-queue.json`
3. 先判断待补原因：
   - 是否当天新入库
   - 是否前端无质保时间显示
4. 执行：
   - `bash scripts/run_scheduled_task.sh sn-warranty-backfill`
5. 读取：
   - `apps/inventory-sync/artifacts/latest-warranty-check-queue.json`
   - `apps/inventory-sync/artifacts/latest-lenovo-warranty-snapshot.json`
6. 核查三件事：
   - 已采成功 SN 是否被排除出待补队列
   - 已有质保时间是否被保留下来
   - 失败或验证码记录是否只保留为待补，不覆盖成功结果

### 训练验收标准

1. 能明确区分“全量重采”和“缺口补齐”。
2. 能说明为什么某个 SN 仍在队列里。
3. 能确认已采成功 SN 不再回到队列。
4. 能确认已有质保起止时间不会被空结果覆盖。
5. 遇验证码、白屏、登录失效时，只能写阻塞，不得改写已有成功结果。

### 常见问题

1. 已采过的 SN 又回到待补：
   - 先查 `latest-lenovo-warranty-snapshot.json`
   - 再查 SQLite `serial_item`
   - 队列生成器只能根据“前端是否仍无质保时间显示”来补缺，不允许全量重置
2. 新一轮采集失败把旧质保时间抹掉：
   - 这是错误行为
   - 失败、验证码、空白结果只能保留阻塞状态，不能覆盖已有 `official_warranty_start / official_warranty_end`
3. 服务端重启后前端又显示待补：
   - 必须先核对 SQLite 是否仍有质保字段
   - 如 JSON 与 SQLite 不一致，以 SQLite 持久层为准重灌
