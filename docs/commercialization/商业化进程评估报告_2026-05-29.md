## 四、第一性原理分析

### 4.1 商业化的本质是什么？

商业化 = **把门店经营数据采集+分析+呈现的能力**，包装成可复制的产品，卖给其他 3C/电脑门店。

三个根本问题：
1. **数据从哪来** → 采集层（智店通、京东、群报价、公众号）✅ 基本成型
2. **数据怎么存** → 数据架构层 ❌ 多租户未落库
3. **数据怎么用** → 前端展示层 ⚠️ 有 mock 过渡

### 4.2 加法：必须新增的功能

```
P0（试点版门槛）：
1. tenant / store / user / role / permission 表 + 基础权限链
2. 核心业务表（sku/serial_item/inventory_movement/sales_order）补 tenant_id/store_id
3. 前端 mock 全清，改成强制读 API
4. 编写第一版 OpenAPI 文档（哪怕是内部版）

P1（试点版可用）：
5. 正式采购入库数据导入跑通
6. 新租户开通向导（门店初始化 → 数据导入 → 正式开卖）
7. 备份恢复 SOP + 实施包
8. snapshot_cache 数据推进到正式历史表
```

### 4.3 减法：可以砍掉或推迟的功能

```
可推迟：
1. 连锁复制版的多层级管理架构（还不是这个阶段）
2. 完整的营销活动历史分析（当前快照镜像够用）
3. 对外 API 市场 / 开发者文档（内部对接够用再对外）
4. 电子价签（汉朔对接）—— 营销加分项但不是试点门槛
5. AI 语音助手（ad-machine 模块）—— 噱头大于实际门店需求
6. 多平台统一价签管理（单一零售场景暂不需要）
```

### 4.4 核心建议：一个中心，三个基本点

**中心**：先把"单门店商业化"跑通，不要同时拉多租户和连锁。

**三个基本点**：

1. **数据链收口**（优先补 tenant_id/store_id + mock 全清）
2. **出入库金额快照收口**（清掉 sync_gap_queue 的 30 条 open 缺口）
3. **最小交付包**（OpenAPI 文档 + 备份恢复 SOP + 开通向导）

## 七、总结

**现状判断**：系统骨架完整，采集层和基础展示层可用，但数据架构和商业交付层缺口明显。当前更像一个"功能验证过的自用工具"，不是"可交付的商业产品"。

**最大风险**：多租户权限层几乎是空白，直接影响商业化最基础的客户隔离能力。

**最小可行路径**：先清 mock → 补多租户基础表 → 跑通采购入库 → 写 OpenAPI 文档 → 做开通向导。按此顺序，21 分的缺口可以在一到两个月内补到 70 分。
