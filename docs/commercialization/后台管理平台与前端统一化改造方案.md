# 联想智慧零售系统后台管理平台与前端统一化改造方案

更新时间：2026-05-24

## 1. 结论

当前系统下一阶段的主任务，不是继续补更多展示卡片，而是把已经存在的管理能力收口成一个真正可操作的后台管理平台，并把前端驾驶舱中重复、混排、样式层级不统一的问题拆开。

系统现状不是“没有后台”，而是“后台能力已经长出来了，但被混在单页驾驶舱里”。

## 2. 当前审计判断

### 2.1 已经存在的后台能力

从代码看，后端和前端已经具备一批典型后台能力，只是入口和版面组织还不对。

- 后端 API 已有：
  - 销售单写入：`POST /api/sales/orders`
  - 采购单写入：`POST /api/purchases/orders`
  - 手工价格覆盖：`/api/inventory-quote/manual-overrides`
  - 库存调整：`/api/inventory-quote/inventory-adjustments`
  - 定时任务控制台：`/api/scheduled-task-console/*`
  - 零售核心查询：`/api/retail-core/sales-orders`、`/order-sync-registry`、`/sync-gap-queue`
- 前端页面里已存在的管理区：
  - 产品库管理
  - 采购入库录入
  - 同步任务与失败队列
  - 定时任务控制台
  - 采集链、规则链、价格校准链

### 2.2 现在真正的问题

不是颜色问题，而是信息架构和组件组织问题。

1. 驾驶舱和后台操作台混在一起  
   现在 `apps/web-cockpit/src/App.tsx` 同时承担：
   - 经营驾驶舱
   - 数据维护台
   - 产品库后台
   - 任务控制台
   - 采购录入台
   - 会话看板

2. 单文件过大，重复区很容易继续长  
   `App.tsx` 已经承载大量书签、子书签、表单、表格、卡片和状态逻辑，后续继续在这里加模块，重复问题只会更严重。

3. 同一类页面缺少统一模板  
   现在不同板块经常各写一套：
   - 顶部摘要区
   - 子标签区
   - 列表区
   - 详情区
   - 操作表单区
   - 状态提示区

4. “看板页”和“维护页”目标不同，但目前视觉层级未彻底分开  
   驾驶舱应该偏“看、查、审计、告警”。  
   后台管理平台应该偏“编辑、导入、导出、批量处理、回放、同步控制”。

## 3. 建议的新产品结构

建议拆成两层，而不是继续所有内容都放在同一个驾驶舱内。

### A. 门店经营驾驶舱

定位：给门店经营、店长、运营查看。

保留这些能力：

- 库存总览
- SN / 保修 / 陈旧库存
- 出入库看板
- 零售报价与价保
- 营销活动与教育补
- 来源价格监控
- 异常提醒

特点：

- 以读为主
- 少量轻操作
- 强调告警、趋势、追溯

### B. 后台管理平台

定位：给管理员、实施人员、运营后台使用。

应该承接这些能力：

- 商品主档 / SKU / 标题 / 分类维护
- 价格规则与采集规则维护
- 链接来源维护
- 手工价格覆盖与库存调整
- 销售 / 采购 / 其他出入库录入与修订
- 订单闭环缺口查看与处理
- 数据导入 / 导出 / 回放
- 本地同步任务 / 失败队列 / 重试
- 定时任务控制台
- 系统配置 / 门店配置 / 后续租户配置

特点：

- 以操作为主
- 支持批量处理
- 支持导入导出
- 支持错误修复和审计

## 4. 后台管理平台建议模块

建议后台先做成 6 个一级模块。

### 4.1 数据总览

- 今日同步状态
- 缺口订单
- 待补 SN / 待补保修
- 待补价格 / 待补链接
- 定时任务状态
- 最近失败队列

### 4.2 商品与价格中心

- 产品主档
- SKU 档案
- 价格校准
- 采集规则
- 链接来源
- 标题与分类修正

### 4.3 库存与出入库中心

- 采购入库录入
- 销售出库查询
- 其他出库处理
- 库存流水
- SN 台账
- 订单闭环缺口处理

### 4.4 营销与价保中心

- 营销活动库
- 价保历史
- 教育补与 PO
- 历史冻结记录
- 来源证据与导出

### 4.5 同步与自动化中心

- 本地同步任务
- 失败队列
- 定时任务控制台
- watchdog
- 手工回放与重建

### 4.6 系统与实施中心

- 数据导入
- 数据导出
- 初始化模板
- 备份与恢复
- API 对接配置
- 门店 / 后续租户配置

## 5. 前端统一化改造原则

### 5.1 先统一页面骨架

所有后台页统一成同一套骨架：

- 页面标题区
- 顶部 KPI 区
- 筛选工具条
- 主列表区
- 右侧详情或下方详情
- 操作抽屉 / 编辑面板

### 5.2 先统一交互组件，不先改配色

当前优先级不是换颜色，而是统一：

- Tabs
- 列表表头
- 表格密度
- 空状态
- 错误态
- 成功/阻塞状态标签
- 表单布局
- 详情面板

### 5.3 驾驶舱与后台风格分层

- 驾驶舱：强调看板、告警、趋势、摘要
- 后台：强调列表、过滤、批量、编辑、导出

不要再让同一版面同时承担“展示”和“大量维护表单”。

## 6. 代码层面建议

### 6.1 当前最大结构问题

`apps/web-cockpit/src/App.tsx` 过大，已经不适合继续承接后台演进。

### 6.2 下一步应该做的拆分

优先拆成：

- `src/features/cockpit/*`
- `src/features/admin-overview/*`
- `src/features/product-library/*`
- `src/features/movements-admin/*`
- `src/features/marketing-admin/*`
- `src/features/sync-console/*`
- `src/features/system-admin/*`
- `src/components/layout/*`
- `src/components/data-display/*`
- `src/components/forms/*`

### 6.3 路由建议

建议引入明确路由，而不是继续所有内容塞进顶层书签：

- `/cockpit`
- `/admin`
- `/admin/products`
- `/admin/movements`
- `/admin/marketing`
- `/admin/sync`
- `/admin/system`

## 7. 推荐执行顺序

### 第一步：先做后台管理平台外壳

目标：

- 独立出 `/admin`
- 左侧导航
- 统一页面容器
- 统一列表页模板
- 统一详情/编辑抽屉模板

这一阶段先不大改业务逻辑，只搬现有能力。

### 第二步：先迁 3 个最有价值的后台模块

优先迁：

1. 商品与价格中心
2. 库存与出入库中心
3. 同步与自动化中心

原因：

- 这三块最直接支撑查看、编辑、导入导出、分析
- 也是商业化后台最像“客户愿意付费的管理台”的部分

### 第三步：清理驾驶舱里的重复维护区

原则：

- 驾驶舱保留查看型能力
- 重操作区迁到后台
- 同一能力只保留一个主入口

### 第四步：统一页面模板和组件

收口：

- 统计卡
- 过滤条
- 状态标签
- 表格
- 详情侧板
- 表单区

## 8. 对当前商业化进度的影响

这项改造会直接提升这些分项：

- 产品完整度
- 商业交付能力
- 文档与实施
- API 与集成能力

它不是“美化工作”，而是商业化收口工作。

## 9. 下一步建议

下一轮直接进入代码改造，顺序建议固定为：

1. 先抽出 `/admin` 后台壳子和统一布局
2. 先把 `产品库`、`出入库管理`、`同步控制台` 从驾驶舱中迁出
3. 再清理驾驶舱里的重复区块和混合操作区
4. 最后统一页面模板、组件和交互密度
