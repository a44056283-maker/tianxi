## 1. 页面定位

`门店商品` 不是经销商总商品池，也不是仓库商品视图。

它在当前项目中的业务定位是：

1. 门店商品知识库
2. 门店侧商品展示和销售状态管理页
3. 后续每日门店实际零售价同步的主要商品落脚页
4. 收银前端商品启停、显示口径、销售属性的门店配置页
5. 我们智慧零售系统的标准 SKU 基准库
6. 门店商品实际零售价的实操更新页

补充固定理解：

1. 后续 `群报价`
2. 后续 `公众号报价`
3. 后续其它报价来源分类

都默认优先拿 `门店商品` 做标准 SKU 匹配基准。

再补一条固定理解：

4. 这页不是纯查看页，而是后续门店零售价改价更新的实际操作页。

当前结论：

> 后续“每日门店售价同步”，默认先围绕 `门店商品` 页面做映射、训练和审计。

## 4. 人工操作教程

以下是给人和 OpenClaw 都能用的标准步骤。

### 4.1 列表页审计步骤

1. 在左侧一级菜单点击 `商品`
2. 在二级菜单点击 `门店商品`
3. 确认地址栏为：
   - `/lenovo/web/product/list-store`
4. 先确认顶部筛选项是否完整出现：
   - 门店
   - 商品分类
   - 商品信息
   - PN/MTM
   - 69码
   - 商品类型
   - 商品状态
5. 再确认按钮区是否出现：
   - 搜索
   - 批量上架
   - 批量下架
   - 导出
6. 再确认标签是否存在：
   - 联想商品
   - 自有商品
   - 全部
7. 再确认表格列是否完整
8. 最后确认每行是否至少有：
   - 查看
   - 编辑

### 4.2 搜索功能手动操作步骤

这一段必须单独讲清，不能让 OpenClaw 自己猜。

这一段以后是高频主入口。

因为后续最常见的动作不是全表翻页，而是：

1. 顶部单品检索
2. 命中单个商品
3. 进入编辑
4. 改价
5. 保存

所以以后默认先教会、先训练、先复用的是“顶部单品检索链”。

#### 搜索用途

`门店商品` 的搜索主要用于：

1. 在标准 SKU 库里定位商品
2. 给群报价匹配商品基准
3. 给公众号报价匹配商品基准
4. 给后续门店售价同步定位目标商品

#### 顶部单品检索是以后最常用的链路

固定理解：

1. 如果目标是改单个商品价格，优先走顶部单品检索
2. 不优先靠翻页找商品
3. 不优先靠肉眼扫全表
4. 检索命中后，直接进入 `编辑`

### 4.2.1 顶部单品检索标准教程

这部分是高频教程，后续 OpenClaw 必须优先复用。

#### 单品检索优先字段

顶部单品检索固定优先级：

1. `商品信息`
2. `PN/MTM`
3. `69码`

如果这三个都不用，就不能说自己做了标准检索。

下面三个字段必须分开学，不能混着用。

### 4.2.2 `商品信息` 检索教程

这是以后使用频率最高的检索框。

#### 这个框用来干什么

1. 查商品编码
2. 查商品名称
3. 查商品简称
4. 快速改单前定位单品

#### 什么时候优先用这个框

1. 已知商品编码时，先用它
2. 已知商品名称或简称时，先用它
3. 不确定 PN/MTM 是否准确时，先用它

#### 这次训练里要输入什么

固定练这两种：

1. `10006834`
2. `联想小新双接口固态闪存盘 XS730`

#### 标准动作

1. 点击顶部 `商品信息` 输入框
2. 清掉旧内容
3. 输入目标值
4. 点击 `搜索`
5. 等表格刷新
6. 核对目标行是否出现
7. 核对商品编码是否正确
8. 核对商品分类、商品类型、状态、销售渠道

#### 命中成功的标准

1. 表格里出现目标商品
2. 商品编码正确
3. 商品分类、类型、状态不冲突

#### 常见错误

1. 输入太长整段商品全名，导致检索不稳定
2. 上次检索内容没清掉
3. 结果出来后只看名字，不核对商品编码

### 4.2.3 `PN/MTM` 检索教程

这是标准型号精确检索框。

#### 这个框用来干什么

1. 用标准型号定位单品
2. 做型号级精确匹配
3. 做价格同步前的型号复核

#### 什么时候优先用这个框

1. 已知准确 PN/MTM
2. 商品名称太长或模糊时
3. 要避开同名近似商品时

#### 这次训练里要输入什么

固定输入：

1. `QXB1T18370`

#### 标准动作

1. 点击顶部 `PN/MTM` 输入框
2. 清掉旧内容
3. 输入 `QXB1T18370`
4. 点击 `搜索`
5. 等表格刷新
6. 核对命中结果是否还是商品编码 `10006834`

#### 命中成功的标准

1. 目标商品被精确筛出
2. 商品编码仍然是 `10006834`
3. 不是别的闪存盘或近似型号

#### 常见错误

1. 把 PN/MTM 输入到 `商品信息`
2. 少字母或多空格
3. 没核对结果是不是同一商品

### 4.2.4 `69码` 检索教程

这是条码精确检索框。

#### 这个框用来干什么

1. 用条码做最精确匹配
2. 排除名称相似商品
3. 条码驱动改单或核品

#### 什么时候用这个框

1. 已知真实 69 码时才用
2. 不知道 69 码时不要乱填
3. 当前样本如果没有确认过 69 码，就不要假装会用

#### 当前教程边界

1. 这次样本 `10006834 / QXB1T18370` 的 69 码在当前页面显示为 `-`
2. 所以本轮只能教会这个框的用途和点法
3. 不能假装本轮已经用 69 码完成过命中

#### 标准动作

1. 点击顶部 `69码` 输入框
2. 清掉旧内容
3. 输入真实 69 码
4. 点击 `搜索`
5. 等表格刷新
6. 核对结果是否唯一命中

#### 常见错误

1. 没有真实 69 码就硬填
2. 把条码当商品编码用
3. 条码命中后不再核对商品信息

### 4.2.5 `商品状态` 检索教程

这是辅助过滤框，不是主检索框。

#### 这个框用来干什么

1. 过滤上架/失效
2. 缩小结果集
3. 辅助改单前确认当前商品是否可售

#### 什么时候用

1. 主检索结果过多时
2. 要确认只看 `上架` 商品时
3. 不单独靠它找商品

#### 标准动作

1. 先完成主检索
2. 再点 `商品状态`
3. 选择需要的状态
4. 再点 `搜索`

#### 常见错误

1. 一上来只选状态，不填主字段
2. 同时加太多条件把结果锁死

### 4.2.6 `商品分类` 检索教程

这是辅助收窄范围的分类框。

#### 这个框用来干什么

1. 缩小搜索范围
2. 避免同名不同类商品混在一起
3. 给批量人工复核做范围限定

#### 什么时候用

1. 主检索结果很多时
2. 已知商品大类时
3. 不单独靠它找单品

#### 标准动作

1. 先完成主检索
2. 再点 `商品分类`
3. 选择正确分类
4. 再点 `搜索`

#### 常见错误

1. 分类选错，导致“查不到”
2. 还没检索单品就先乱收窄范围

#### 单品检索适用场景

1. 已知商品编码，要快速定位改单
2. 已知 PN/MTM，要快速定位单品
3. 已知 69 码，要做条码精确命中
4. 已知商品名或简称，要做名称检索后改单

#### 单品检索标准动作

1. 进入 `商品 -> 门店商品`
2. 确认当前页签是 `联想商品`
3. 先确认 `门店` 已经是当前正确门店
4. 清空上一次残留搜索条件
5. 根据目标信息选字段：
   - 有商品编码或名称：用 `商品信息`
   - 有 PN/MTM：用 `PN/MTM`
   - 有 69 码：用 `69码`
6. 只填一个主检索字段，先不要乱加多个条件
7. 点击 `搜索`
8. 等表格刷新
9. 核对结果列：
   - 商品信息
   - 商品分类
   - 商品类型
   - 商品编码
   - 状态
10. 命中后直接点击 `编辑`

#### 单品检索示范样本

本轮已验证样本可以直接拿来做训练：

1. 商品编码：`10006834`
2. SKU编码：`20007533`
3. PN/MTM：`QXB1T18370`
4. 商品名称：`联想小新双接口固态闪存盘 XS730 活力橙（256GB）`

对应检索练习顺序：

1. 先用 `商品信息 = 10006834`
2. 再用 `PN/MTM = QXB1T18370`
3. 最后再用 `商品信息 = 联想小新双接口固态闪存盘 XS730`

这三种都要练，因为这是以后最常见的改单入口。

#### 单品检索命中后的判断规则

命中商品不能只看名字像。

至少要同时核对：

1. 商品编码
2. 商品分类
3. 商品类型
4. 状态
5. 销售渠道

只要这些信息不一致，就不能直接进入改价。

#### 单品检索常见错误

1. 上一次检索条件没有清掉，导致结果异常
2. `商品信息` 里输入太长全名，结果反而不稳定
3. 一开始就同时填 `商品信息 + PN/MTM + 商品状态`，把结果锁死
4. 没切到 `联想商品` 就开始检索
5. 命中后不核对列信息，直接误点错误商品

#### 当前可见搜索字段

1. `门店`
2. `商品分类`
3. `商品信息`
4. `PN/MTM`
5. `69码`
6. `商品类型`
7. `商品状态`

#### 推荐搜索顺序

固定按下面顺序理解，不要乱搜：

1. 先用 `商品信息`
   - 适合商品名称、简称、商品编码
2. 再用 `PN/MTM`
   - 适合标准型号匹配
3. 再用 `69码`
   - 适合条码精确匹配
4. 再配合 `商品状态`
   - 过滤上架、已失效
5. 再配合 `商品分类`
   - 收窄范围

#### 手动点击搜索标准步骤

1. 进入 `商品 -> 门店商品`
2. 先确认 `门店` 字段是否已经落在正确门店
3. 如果要按名称或商品编码查：
   - 点击 `商品信息`
   - 输入目标名称、简称或商品编码
4. 如果要按标准型号查：
   - 点击 `PN/MTM`
   - 输入目标 PN/MTM
5. 如果要按条码查：
   - 点击 `69码`
   - 输入目标 69 码
6. 如果要过滤状态：
   - 点击 `商品状态`
   - 选择目标状态
7. 条件填好后，再点击 `搜索`
8. 等表格刷新后，再逐列判断结果

#### 搜索结果怎么看

至少看这几列：

1. `商品信息`
2. `商品来源`
3. `商品分类`
4. `商品类型`
5. `商品编码`
6. `状态`
7. `销售渠道`

如果只是名字像，但分类、类型、状态不对，不能算匹配成功。

5. 再看 `规格信息` 区
6. 核对是否出现：

### 4.8 本次改价过程中遇到的实际操作问题

这部分必须写清楚，后面训练 OpenClaw 也必须按这个问题清单处理，不能只写成功路径。

问题 1：`Computer Use` 能看到步进器，但直接写值失败。

真实表现：

1. 在无障碍树里可以看到：
   - `步进器 (settable, float) Value: 499, Details: 499.00¥`
2. 但直接对该元素做 `set_value` 失败
3. 失败信息是：
   - `Computer Use server error -10005: ... is an invalid element ID`

结论：

1. 不是页面没有价格控件
2. 是 `Computer Use` 对这个 Ant Design 步进器控件的元素写入不稳定

问题 2：选中 `499.00¥` 后，键盘直接输入不稳定。

真实表现：

1. 可以视觉上选中 `499.00¥`
2. 但直接 `type_text` 或 `BackSpace` 不稳定
3. 页面值不会可靠地跟着变化

结论：

1. 不是登录态问题
2. 不是权限问题
3. 是这个控件在当前无障碍链路下，对纯键盘替换支持不好

问题 3：视觉点击步进器上下箭头，命中率不稳定。

真实表现：

1. 页面缩放为 `67%`
2. 步进器上下箭头区域很小
3. 单纯靠坐标点上下箭头，容易命中空白或文本区

结论：

1. 小控件不适合只靠肉眼坐标连续试点
2. 需要更稳定的控件定位方式
