# OpenClaw 销售 / 采购 / 库存边界整改指令

更新时间：2026-05-31  
执行对象：OpenClaw 智店通采集 CLI / PostgreSQL `zdt_sync`  
目的：把 OpenClaw 采集结果整改成可被智慧零售系统稳定消费的 SQL 主链，消除“销售出库 / 采购入库 / 库存详情”混类、缺价格、缺 SN、金额口径错乱的问题。

---

## 1. 这次必须解决的不是展示问题，而是源头问题

当前零售系统前端暴露出的错乱，根因不在前端，而在 OpenClaw 源库输出边界不清：

1. `fact_orders` 同一业务单重复落库，消费端只能反复去重。
2. `fact_order_items` 存在大量壳明细，只保留了 `sku_no`，没有商品名、SN、金额。
3. 金额字段口径混乱：`pay_amount` 常是元，`unit_price / total_amount` 常是分。
4. 销售、采购、退货、调拨、其他出入库没有做稳定的业务分类字段，导致消费端只能靠单号前缀和备注猜。
5. 库存类数据没有形成“库存快照 + SN 在库 + 入库来源 + 供应商”的完整闭环。

这会直接造成：

1. 采购入库被混进销售出库。
2. 销售出库实付金额显示为 0 或单位错误。
3. 库存详情显示有库存，但缺成本、缺规格、缺 SN 明细。
4. 消费端每次同步后都要二次修补，无法形成稳定自动链路。

---

## 2. 当前已审计到的明确问题

以下问题来自 `postgresql://zdt:zdt@localhost:5432/zdt_sync` 现库审计：

### 2.1 重复订单问题

- `fact_orders` 中 `order_no` 重复分组数：`787`

这说明当前订单头不是稳定一单一头，消费端无法直接把它视为销售主链。

### 2.2 壳明细问题

- `fact_order_items` 中满足以下条件的壳明细共有：`896`
  - `pay_amount = 0`
  - `total_amount = 0`
  - 且 `product_name / sku_no / serial_number` 至少一个为空

典型样本：

- `order_no = NULL, sku_no = 20007932, serial_number = ''`
- `order_no = NULL, sku_no = 10007280_20007980, serial_number = ''`

这种行不能当正式订单明细落正式事实表，只能当原始未解结构材料单独存放。

### 2.3 金额单位混乱问题

- `fact_order_items` 中 `pay_amount` 与 `total_amount` 比值明显异常的行数：`885`

典型样本：

- `XS26010461908099861`: `pay_amount=31998.00`, `unit_price=1599900.00`, `total_amount=3199800.00`
- `XS25123072423452751`: `pay_amount=55992.00`, `unit_price=699900.00`, `total_amount=5599200.00`

这证明当前表中至少存在：

1. `pay_amount` 用元
2. `unit_price / total_amount` 用分

消费端可以兜底，但不应该长期依赖兜底。

### 2.4 业务分类边界不清

当前零售系统已经实页看到：

1. `ZDT-CGR...` 采购单仍会出现在“销售出库”页。
2. 采购入库虽然能显示，但很多行仍只能显示“进货成本待补”。
3. 库存详情明明有 SN 数量，但行级细节无法稳定使用源数据补齐。

这说明 OpenClaw 输出还缺少稳定的：

1. `business_scope`
2. `source_document_type`
3. `document_direction`
4. `canonical_document_no`

## 3. 必须新增或整改的字段规范

## 3.1 订单头 `fact_orders` / 新标准订单头

必须补齐或统一输出：

- `order_id`
- `order_no`
- `canonical_order_no`
- `business_scope`
- `source_document_type`
- `document_direction`
- `order_type_name`
- `channel_type_name`
- `status`
- `status_name`
- `shop_id`
- `shop_name`
- `customer_name`
- `customer_phone`
- `cashier_name`
- `total_quantity`
- `total_amount_yuan`
- `pay_amount_yuan`
- `business_time`
- `updated_time`
- `raw_payload`

### `business_scope` 允许值必须收敛为：

- `sales_outbound`
- `purchase_inbound`
- `sales_return_inbound`
- `transfer_outbound`
- `transfer_inbound`
- `other_outbound`
- `other_inbound`
- `same_store_location_change`

禁止继续让消费端去猜。

## 3.2 订单明细 `fact_order_items` / 新标准订单明细

必须补齐或统一输出：

- `order_item_id`
- `order_id`
- `order_no`
- `canonical_order_no`
- `business_scope`
- `sku_no`
- `product_no`
- `product_name`
- `mtm_code`
- `spec`
- `serial_number`
- `quantity`
- `unit_price_yuan`
- `total_amount_yuan`
- `pay_amount_yuan`
- `discount_amount_yuan`
- `source_document_type`
- `business_time`
- `raw_payload`

### 明细硬规则

1. 金额字段统一输出为“元”。
2. 壳明细不得进入正式事实表。
3. 壳明细必须进入单独原始缓冲表，例如：
   - `fact_order_items_raw_unresolved`
4. 没有 `order_no` 的明细不能直接进入正式销售事实表。

## 4. 库存与 SN 必须补齐的事实链

必须保证零售系统能直接消费以下闭环：

1. 当前库存数量
2. 当前可售数量
3. 当前在库 SN 列表
4. 每个 SN 的入库时间
5. 每个 SN 的入库单号
6. 每个 SN 的供应商
7. 每个 SKU 的平均进货成本

最低需要稳定输出：

- `inventory_snapshot`
- `sn_records`
- `purchase_orders`
- `purchase_order_details`
- `suppliers`

并保证这些表之间能通过以下键稳定关联：

- `sku_no`
- `serial_number`
- `purchase_order_no`
- `supplier_id / supplier_name`

---

## 5. 业务边界硬规则

### 5.1 销售出库

只允许进入销售主链的单据：

- `XS...`
- 明确为 `业务订单 / 门店出库 / 待商确认`

### 5.2 采购入库

只允许进入采购主链的单据：

- `CGR...`
- 明确为 `采购入库`

### 5.3 禁止混类

以下单据不得再写入销售主链：

- `ZDT-CGR...`
- `PURCHASE-CGR...`
- `PURCHASEQ-CGR...`
- 退货入库
- 调拨入库
- 同店换库位入库

消费端已经被迫写了兜底规则，但这不是长期方案。源头必须直接输出正确边界。

## 6. OpenClaw 具体整改步骤

### P0：先修正式事实表口径

1. 对 `fact_orders` 去重，按 `canonical_order_no` 保证一单一头。
2. 对 `fact_order_items` 去壳，正式表只留可消费明细。
3. 所有金额字段统一转为“元”。
4. 给订单头与订单行都增加 `business_scope`。
5. 给订单头与订单行都增加 `source_document_type`。
6. 给订单头与订单行都增加 `canonical_order_no`。

### P1：补采购与库存消费所需实体

1. `purchase_orders`
2. `purchase_order_details`
3. `inventory_snapshot`
4. `sn_records`
5. `suppliers`

### P2：补异常与缓冲层

1. 壳数据缓冲表
2. 原始留痕表
3. `sync_state` 精确到实体级
4. 失败重试与人工复核队列

---

## 7. 验收标准

整改完成后，必须同时满足：

1. `fact_orders` 中同一 `canonical_order_no` 不重复。
2. `fact_order_items` 正式表不再存在 0 金额壳明细。
3. `fact_order_items` 金额字段全部统一为元。
4. 销售、采购、退货、调拨在 SQL 层就已分类清楚。
5. 零售系统前端中：
   - `销售出库` 不再出现 `CGR` 单
   - `采购入库` 不再混入销售单
   - `库存详情` 能自动补齐成本、规格、SN 明细
6. 零售系统不再需要通过备注、前缀、页面启发式去猜业务类别。
