# ZDT 商品/订单/库存全量采集审计与对接方案

更新时间：2026-05-31  
审计范围：
- OpenClaw ZDT PostgreSQL：`postgresql://zdt:zdt@localhost:5432/zdt_sync`
- 智店通真实页面：商品、订单、库存三大主菜单及其子菜单
- 本项目零售系统进销存对接链：`apps/inventory-sync`、`apps/api-server`、`apps/web-cockpit`

---

## 1. 结论

当前 OpenClaw 版 ZDT 数据库不是“没数据”，而是“采得不全、实体不齐、字段不够、表意不统一”。

已经有基础数据的表：

- `fact_products`
- `fact_orders`
- `fact_order_items`
- `fact_inventory`
- `fact_stock_orders`
- `fact_sn_records`
- `fact_purchase_orders`
- `fact_purchase_order_details`

当前主要缺口不是一张表，而是三类缺口同时存在：

1. 商品侧缺口：缺门店商品/仓库商品发布态、收银热卖商品、供应商主数据、税率与 69 码等字段。
2. 订单侧缺口：缺退单链、线上订单专有字段、OMS/渠道订单号、退款单明细。
3. 库存侧缺口：缺库位库存快照、库存总览快照、出入库单据头/单据行、销售成本价、同店换库位、调拨出入库等业务单实体。

因此当前不能只靠已有 `fact_*` 表就宣称“智店通商品/订单/库存三大类已完整对接”。  
必须把 OpenClaw 采集链补成“商品主数据 + 订单事实 + 库存事实 + 出入库单据 + 发布态 + 供应商 + 成本价”这一整套。

## 4. OpenClaw 当前数据库与真实页面的缺口矩阵

## 4.1 商品侧

### 已有

- `fact_products`：商品编码、商品名、分类文本、状态、来源、类型、更新时间

### 缺失

1. `69码`
2. `PN/MTM`
3. 税率分类 / 税率
4. 门店商品发布态（上架/下架、销售渠道）
5. 仓库商品发布态
6. 收银热卖商品清单与排序
7. 供应商主数据
8. 商品图片之外的门店/仓库作用域

### 结论

`fact_products` 只能算“经销商商品基础主数据”，不能代表完整商品链。

## 4.2 订单侧

### 已有

- `fact_orders`
- `fact_order_items`
- `fact_order_payments`

### 缺失

1. 线下退单头表 / 明细表
2. 线上退单头表 / 明细表
3. 线上订单专有字段：
   - `OMS订单号`
   - `渠道订单号`
   - `OMS订单状态`
   - `订单渠道`
4. 退单字段：
   - 退款单号
   - 退款单类型
   - 申请退款金额
   - 是否退货
   - 申请人
   - 申请数量
   - 退单状态

### 结论

当前订单链只覆盖了“订单成交事实”，没有覆盖“退单/售后回流事实”。

## 4.3 库存侧

### 已有

- `fact_inventory`
- `fact_stock_orders`
- `fact_sn_records`
- `fact_purchase_orders`
- `fact_purchase_order_details`

### 缺失

1. 库位库存快照
2. 库存总览快照
3. 库存订单行表（当前 `fact_stock_order_items = 0`）
4. 销售成本价事实表
5. 调拨出库单头 / 单行
6. 调拨入库单头 / 单行
7. 其他出入库单头 / 单行
8. 同店换库位单头 / 单行
9. 商品库存页里的销售成本记录
10. 库存水位预警配置

### 结论

当前库存链只能支撑“库存快照 + 流水简表 + SN 流转 + 采购入库”。  
对门店真实经营必须补齐“库位、单据、成本、调拨、其他出入库”。

## 5. 目标表设计（OpenClaw ZDT SQL）

建议不要再把所有内容继续硬塞进现有几张 `fact_*` 表。应补成下列目标表。

## 5.1 商品

1. `fact_product_master`
   - 商品基础主数据
   - 来源：经销商商品

2. `fact_product_publish_store`
   - 门店商品发布态
   - 字段：
     - `shop_id`
     - `product_no`
     - `status_name`
     - `sale_channel`
     - `updated_time`

3. `fact_product_publish_depot`
   - 仓库商品发布态

4. `fact_hot_sale_products`
   - 收银热卖商品
   - 字段：
     - `shop_id`
     - `product_no`
     - `sort_index`
     - `is_combo`
     - `is_on_shelf`

5. `dim_suppliers`
   - 供应商主数据

## 5.2 订单

1. `fact_orders_offline`
2. `fact_order_items_offline`
3. `fact_orders_online`
4. `fact_order_items_online`
5. `fact_order_refunds_offline`
6. `fact_order_refund_items_offline`
7. `fact_order_refunds_online`
8. `fact_order_refund_items_online`

说明：
- 若继续共用 `fact_orders` / `fact_order_items`，也必须增加 `entity_scope` 或 `business_scope` 明确区分。
- 退单不可只塞进 `raw_payload`，必须有独立结构化表。

## 5.3 库存

1. `fact_inventory_snapshot`
2. `fact_inventory_location_snapshot`

4. `fact_stock_stream`

10. `fact_transfer_out_lines`

## 5.4 原始留痕

1. `raw_records`
   - 必须恢复成 append-only 原始页面/接口回执留痕
   - 至少包含：
     - `entity_name`
     - `source_url`
     - `payload_json`
     - `collected_at`
     - `hash_key`

2. `sync_state`
   - `entity_name` 命名必须统一
   - 不能再混用 `order / orders / orders_online`

3. `sync_job`
   - 每次采集任务必须写入起止时间、状态、行数、异常摘要

## 7. OpenClaw 定时采集更新流程

## 7.1 商品链

### 高频轻量

每 30 分钟：

1. 门店商品
2. 仓库商品
3. 收银热卖商品
4. 供应商管理

用途：
- 更新发布态
- 更新收银热卖排序
- 更新供应商主数据

### 低频全量

每天 2 次：

1. 经销商商品全量

用途：
- 商品主数据基表
- 69码 / 分类 / 税率 / PN-MTM 基础维护

## 7.2 订单链

每 30 分钟：

1. 线下门店订单
2. 线下门店退单
3. 线上订单
4. 线上退单

采集顺序：

1. 先采单头列表
2. 再采明细
3. 再采支付/退款补充字段
4. 最后写 `sync_state` 和 `sync_job`

## 7.3 库存链

每 30 分钟：

1. 商品库存
2. 库位库存
3. 库存流水
4. 库存订单
5. SN库存订单

## 10. 自动同步闭环流程（ZDT SQL -> 零售系统）

目标不是“OpenClaw 采完就算完”，而是形成进销存三大类自动收口。

## 10.1 商品闭环

1. OpenClaw 把 `products_master / products_store_publish / products_depot_publish / products_hot_sale / suppliers` 写入 ZDT PostgreSQL
2. 零售系统 `apps/inventory-sync` 新增 `zdt_product_sync` 消费任务
3. 写入本地：
   - `product`
   - `sku`
   - `supplier`
   - 商品发布态投影表
4. `apps/api-server` 读取 SQLite 后更新：
   - 商品库
   - 收银商品投影
   - 热卖排序
   - 供应商筛选
5. 前端自动刷新

验收口径：

- 商品、门店商品、仓库商品、热卖商品、供应商 5 类实体都有最新 `collected_at`
- 前端可见 SKU 数、供应商数与 SQL 同步

## 10.2 订单闭环

1. OpenClaw 把 `orders_* / order_items_* / order_refunds_*` 写入 ZDT PostgreSQL
2. 零售系统 `zdt_order_sync` 按 `order_no + item_id + refund_no` 增量消费
3. 写入本地：
   - `sales_order`
   - `sales_order_line`
   - `customer`
   - `order_sync_registry`
   - `sync_gap_queue`
4. 非线下销售、退单、线上单据分别落到独立子书签
5. 前端按业务时间刷新

验收口径：

- 订单头行数一致
- 退单不再混进销售出库
- 订单详情含客户信息、外部单号、渠道单号、订单状态

## 10.3 库存闭环

1. OpenClaw 把 `inventory_snapshot / inventory_location_snapshot / stock_stream / stock_orders / stock_order_items / sn_records / purchase_orders / other_inout / transfer / same_store_location_change` 写入 ZDT PostgreSQL
2. 零售系统 `zdt_inventory_sync` 增量消费
3. 写入本地：
   - `inventory_movement`
   - `serial_item`
   - `inventory_snapshot`
   - `inventory_location_snapshot`
4. 通过 `service_no / document_no / serial_number / sku_no` 去重合并
5. 前端库存数量、SN 数量、单据流水自动刷新

验收口径：

- 已出库 SN 不再停留在在库
- 已入库供应商能按 SN 和入库单反推
- 库存数量、可售数量、SN 数量与 ZDT SQL 对齐

## 10.4 同步触发规则

建议拆成 3 条同步任务：

1. `zdt-product-master-sync`
   - 每 30 分钟检查一次
   - 有新 `collected_at` 就增量同步

2. `zdt-order-master-sync`
   - 每 10 分钟检查一次
   - 按 `pay_time / collected_at` 双游标推进

3. `zdt-inventory-master-sync`
   - 每 10 分钟检查一次
   - 按 `business_time / collected_at` 双游标推进

硬规则：

- 只要 ZDT SQL 有新记录，就自动触发本地 SQLite 更新
- 本地 SQLite 更新成功后，前端 API 轮询或推送自动刷新
- 任何失败都写 `sync_job + sync_state + sync_gap_queue`

## 13. 本轮审计后的直接结论

1. OpenClaw 当前数据库可作为 ZDT SQL 主链雏形，但还不能支撑智慧零售系统完整进销存闭环。
2. 当前最先要补的不是“更多前端页面”，而是 OpenClaw 的实体和字段完整性。
3. 商品、订单、库存三大类必须按本方案拆成结构化表后，零售系统再做自动映射消费，才能真正实现：
   - 智店通采集
   - ZDT SQL 落库
   - 本地 SQLite 同步
   - 前端自动更新
4. 在上述实体补齐前，任何“已完整对接智店通”的说法都不成立。
