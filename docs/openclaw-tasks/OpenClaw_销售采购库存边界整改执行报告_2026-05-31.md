## 1. P0 整改完成情况

### ✅ 1.1 fact_orders 去重

**问题**: 同一 `order_no` 有 2 条记录（有效行 + 壳行），共 810 组重复

**根因**: ZDT 订单列表 API 每个订单返回 2 条（header 聚合行 + 明细行），历史采集脚本未做去重

**处理**:
- 识别 `order_id = order_no` 的 812 行壳行（`status/pay_amount/status_name` 均为空）
- 其中 810 行含有效关联明细 → 先将明细重映射至有效 header，再删除壳行
- 剩余 2 行 `SO-` 壳 header（含有效 item 但无 numeric parent header）→ 保留壳 header 以满足 FK 约束，item 的 `order_id` 置 NULL
- 最终：删除 812 行重复壳 header，保留 821 行有效 header

**结果**:
```

重复 order_no 组数: 0 ✅
```

### ✅ 1.4 金额单位统一为元

**问题**: ZDT API 返回 fen，原始采集脚本部分字段（元/分）未做转换，导致同一行中 `unit_price` 与 `pay_amount` 单位混乱

**处理**:

**unit_price / total_amount（fact_order_items）**:
- 规则：`unit_price >= 10000` → fen → 除以 100
- 转换 967 行

**pay_amount（fact_order_items）**:
- 规则 1：`pay_amount >= 100000` → 明显 fen（如 ¥5799 商品不可能是 579900 元）→ 除以 100 → 转换 27 行
- 规则 2：`pay_amount >= 10000 AND pay_amount / unit_price > 10` → fen → 除以 100 → 转换 12 行
- 剩余 117 行 `pay_amount >= 10000` 且 `ratio ≈ 1`（如 ¥28999 高端笔记本）→ 确认为元，不转换

**pay_amount / total_amount / product_amount（fact_orders）**:
- 规则：`pay_amount >= 100000` → fen → 除以 100
- 转换 700 行

**结果**:
```
转换后 fact_order_items 金额范围:
  pay_amount: [0, 55992] 元
  unit_price: [0, 28999] 元
  所有大额金额均符合实际商品价格 ✅
```

**注**: 整改后仍有个别行金额单位不统一（如 `pay_amount` 与 `unit_price` 混用），属于 API 返回数据本身的口径问题，需在采集层统一元/分规范，禁止混用。

## 4. 遗留问题

### 4.1 fact_orders 仍含 2 个 `SO-` 壳 header
- `SO-20260512-0001`（已完成）和 `SO-20260526-0002`（预留）各有 1 条有效 item
- 为满足 FK 约束（`ON DELETE SET NULL`），保留壳 header，`order_id` 置 NULL
- 注：业务上这是工程软件订单，有实际收款，不属于"壳数据"

### 4.2 金额单位历史数据
- 历史 `fact_order_items` 中仍有少量行 `pay_amount` 与 `unit_price` 单位不一致
- 根本原因：ZDT 同一 API 返回不同字段用不同单位
- 解决：后续采集脚本必须在入库前统一做 `amount_yuan = amount_fen / 100` 转换，禁止混用

### 4.3 FK 约束改为 ON DELETE SET NULL
- 原 `fact_order_items.order_id → fact_orders.order_id` 为严格 FK
- 改为 `ON DELETE SET NULL` 以允许 item 独立存在（header 删除后 item 变孤立但不丢失）
- 建议后续消费端 JOIN 时加 `WHERE order_id IS NOT NULL` 条件
