# ZDT 实时对接数据质量审计

日期：2026-05-31  
范围：OpenClaw/ZDT PostgreSQL `zdt_sync` 与本地 `apps/api-server/data/retail-core.sqlite3` 的只读审计。  
结论状态：已定位主要同步错乱原因，尚未执行修复。

## 一、总判断

ZDT 采集数据不是整体垃圾数据。PostgreSQL 事实层已经有比较完整的数据结构：

- `fact_orders`
- `fact_order_items`
- `fact_purchase_orders`
- `fact_purchase_order_details`
- `fact_stock_orders`
- `fact_sn_records`
- `fact_inventory`

真正造成前端“数量、SN、成本、入库出库混乱”的主要原因是：

1. 本地 SQLite 同步层把不同质量、不同业务粒度的数据混在 `inventory_movement` 一张表里直接给前端使用。
2. 同步脚本存在错误分类：`待商确认` 被映射成 `sales_outbound`。
3. 金额单位没有统一规范化：订单表金额来自 ZDT 是分单位，采购成本是元单位，本地同步只对部分字段做了除以 100。
4. `sales_order` 中存在大量壳单，和 `fact_order_items` / `inventory_movement` 的真实出库事实没有清晰分层。
5. 前端展示层缺少 canonical view，只能靠单据号、movement_type、字段是否为空做推断，容易把采购入库、销售出库、库存流水单混排。

## 二、关键证据

### 1. PostgreSQL 源事实表有数据

只读统计结果：

| 表 | 行数 |
|---|---:|
| `fact_inventory` | 14066 |
| `fact_sn_records` | 3142 |
| `fact_stock_orders` | 1816 |
| `fact_order_items` | 1003 |
| `fact_orders` | 828 |
| `fact_purchase_order_details` | 407 |
| `fact_purchase_orders` | 260 |
| `other_inout_documents` | 0 |
| `other_inout_lines` | 0 |

判断：主链路不是没有数据，采购、库存流水、订单明细、SN 明细都在。`其他出入库` 仍为空，属于另一个缺口。

### 2. 本地 SQLite `inventory_movement` 数据质量分层明显

按来源统计：

| movement_type | source_system | 行数 | 缺 SN | 缺金额 | 缺成本 |
|---|---|---:|---:|---:|---:|
| `sales_outbound` | `system_inventory_movements` | 1265 | 730 | 504 | 1265 |
| `sales_outbound` | `zhidiantong` | 1002 | 26 | 9 | 12 |
| `purchase_inbound` | `system_inventory_movements` | 490 | 300 | 52 | 52 |
| `transfer_outbound` | `system_inventory_movements` | 381 | 290 | 381 | 381 |
| `sales_outbound` | `smart_retail_snapshot` | 251 | 139 | 97 | 251 |

判断：`zhidiantong` 来源比旧 `system_inventory_movements` 明显更完整。问题不是新采集源全烂，而是前端/API 混用了旧快照、手工回填、真实 ZDT 三类来源。

### 3. `待商确认` 被错误写成销售出库

代码位置：

- `apps/inventory-sync/src/storage/zdtSqliteSync.py`
- `apps/inventory-sync/src/storage/syncZdtToLocalSqlite.py`

当前映射：

```python
"待商确认": "sales_outbound"
```

本地 SQLite 已出现错误样例：

| id | 实际单号 | 商品 | 数量 | 当前 movement_type | 当前 service_type |
|---|---|---|---:|---|---|
| `ZDT-CGR260531422330_...` | `CGR260531422330` | razr fold 16+512 | 2 | `sales_outbound` | `销售出库` |
| `ZDT-CGR260531422299_...` | `CGR260531422299` | 小新Pro14 R7-H255 | 10 | `sales_outbound` | `销售出库` |
| `ZDT-CGR260531422229_...` | `CGR260531422229` | YOGA Air 14 ILL10 | 5 | `sales_outbound` | `销售出库` |
| `ZDT-CGR260531422218_...` | `CGR260531422218` | 小新Pro16c U5 | 9 | `sales_outbound` | `销售出库` |
| `ZDT-CGR260531421964_...` | `CGR260531421964` | Y7000P i9 | 6 | `sales_outbound` | `销售出库` |

判断：这是 P0 级别错误。CGR 单据不应该按销售出库展示，也不应该影响销售价保或销售流水。

### 4. 金额单位混乱

PostgreSQL `fact_order_items` 示例：

| 订单 | 商品 | PG unit_price |
|---|---|---:|
| `XS26053145814187319` | Y7000 IRX10 | 949900.00 |
| `XS26053114116386119` | 小新Pro16 R7-H255 | 569900.00 |
| `XS26053178588167619` | 小新 27-IRH | 579900.00 |

本地 SQLite 对应部分字段：

| 字段 | 当前表现 |
|---|---|
| `inventory_movement.amount` | 已被转换成元，例如 9499.0 |
| `inventory_movement.unit_cost` | 仍是分单位，例如 949900.0 |

判断：订单来源金额需要统一转换为元；采购 `fact_purchase_order_details.cost_price` 已经是元，不应再次除以 100。必须按来源表和字段分别定义金额单位。

### 5. `sales_order` 存在大量壳单

本地 `sales_order` 统计：

| order_type | channel | 行数 | 金额为 0 | 无收银员 |
|---|---|---:|---:|---:|
| 线上订单 | 有赞 | 826 | 0 | 0 |
| 线下 | 门店收银 | 362 | 362 | 362 |
| 工程软件直销 | 门店收银 | 2 | 0 | 2 |

判断：`sales_order` 里有 362 条零金额、无收银员的线下壳单。它们不能直接作为“已完整同步销售单”展示，必须标记为 `shell_pending_enrichment` 或用 `fact_order_items` / `inventory_movement` 的真实 XS 单据补齐后再进入前端。

### 6. 采购入库与 SN 对齐仍有缺口

本地对 `purchase_inbound` 按 `doc + sku` 比对：

- 大量 `TDR0000072xx` 行显示 movement 数量为 2，但匹配 SN 为 0 或 1。
- 大量 `PURCHASEQ-*` 临时队列行没有对应 `serial_item`。
- 部分 `CGR` 单据中 movement 数量与 SN 数量不一致。

判断：`TDR/PURCHASEQ/CGR` 需要分层，不能混在同一个采购入库列表里当作同等质量数据。`CGR` 是更接近智店通真实采购/库存流水单的主证据，`PURCHASEQ-*` 更像补录队列，不应该驱动前端真实库存。

## 三、根因归类

### A. 不是垃圾数据，是缺少标准事实层

OpenClaw/ZDT 事实表中已有 `business_scope`、`source_document_type`、`canonical_order_no` 等字段，但同步到 SQLite 后丢失或弱化了这些层次。

建议新增或明确一个 canonical 层：

```text
zdt_raw_*                  原始证据层，保留不改
zdt_normalized_movements   标准流水层
zdt_serial_ledger          标准 SN 台账层
zdt_sales_facts            标准销售事实层
zdt_purchase_facts         标准采购事实层
frontend_projection_*      前端展示投影层
```

### B. 同步器分类规则错误

`待商确认` 当前被当作 `sales_outbound`，直接导致入库数量进了销售出库。

建议：

```text
订单出库       -> sales_outbound
采购入库       -> purchase_inbound
订单退货入库   -> sales_return_inbound
调拨出库       -> transfer_outbound
调拨入库       -> transfer_inbound
同店换库位出库 -> location_transfer_out
同店换库位入库 -> location_transfer_in
门店出库       -> other_outbound
待商确认       -> pending_stock_flow
```

其中 `pending_stock_flow` 不能进入销售价保，也不能算零售销售。

### C. 金额单位必须显式规范化

建议新增字段或 view：

```text
amount_raw
amount_unit: cent | yuan | unknown
amount_yuan
unit_price_raw
unit_price_yuan
cost_price_raw
cost_price_yuan
```

规则：

- `fact_orders.total_amount/pay_amount/product_amount`：分 -> 元。
- `fact_order_items.unit_price/total_amount/pay_amount/discount_amount`：分 -> 元。
- `fact_purchase_order_details.cost_price/batch_cost_price`：元 -> 元。
- `fact_inventory.cost_price`：需抽样确认后再定。

### D. 前端不能再直接吃混合流水

前端 `采购入库 / 销售出库 / 库存详情 / 价保` 应读取标准投影，不应直接把 `inventory_movement` 所有来源混排。

最低要求：

- 销售出库只读 `sales_outbound` 且 `source_quality in ('zdt_fact_order_item','zdt_stock_order_confirmed')`。
- 采购入库只读 `purchase_inbound` 和明确 `CGR` 采购单。
- `pending_stock_flow` 单独放“待确认库存流水”，不参与销售、价保和真实库存扣减。
- `system_inventory_movements` 旧源默认降级为历史证据，不能覆盖 `zhidiantong` 新源。

## 四、P0 修复顺序

1. 修正 `待商确认` 映射，禁止继续写入 `sales_outbound`。
2. 对已写错的 `ZDT-CGR*` 行做一次迁移：
   - 改为 `pending_stock_flow` 或待确认入库分类。
   - 恢复 `source_document_type='待商确认'`。
   - 从销售出库、销售价保、销售流水投影中排除。
3. 修正订单金额同步：
   - `amount`、`unit_cost` 等字段统一用元。
   - 或新增 `*_yuan` 字段，旧字段保留但不再给前端直接使用。
4. 为 `sales_order` 壳单加质量状态：
   - `complete`
   - `shell_pending_enrichment`
   - `legacy_snapshot`
5. 新建 ZDT 数据质量审计脚本，每轮同步后输出：
   - 错分 CGR 数量。
   - 销售出库无 SN 数量。
   - 销售金额异常数量。
   - 采购单数量与 SN 数量不一致清单。
   - `source_system` 混用统计。
6. 前端改读标准投影，并在 UI 上显示“真实 ZDT / 历史证据 / 待确认 / 阻塞”状态。

## 五、当前审计结论

本次问题不是“OpenClaw 采到的全是垃圾数据”。更准确地说：

1. ZDT 原始事实层有大量可用数据。
2. 本地同步层没有建立足够清楚的业务分类和质量分层。
3. 同步器存在明确映射 bug，把 `待商确认` 写成了销售出库。
4. 金额单位规范化不完整，导致销售价、进货价、成本价在前端看起来混乱。
5. 前端展示层没有只读 canonical projection，而是混吃多源流水，所以出现“有些同步了、有些没同步、有些同步错了”的现象。

下一步应先做 P0 修复，不建议继续在 UI 层补丁式隐藏问题。
