# 02 Decisions

## 2026-06-11 教育补代扫正式主链切换为今日相机 OpenAPI，网页分类文件夹只保留为备用 CLI 归集链

- 从现在起，教育补代扫的正式采集主链固定为：
  - `今日相机 OpenAPI -> Bridge API -> education_scan_record_v2 -> latest-education-subsidy-agent-scan-summary.json -> 前端`
- 原先“网页微信群逐图回扫”不再作为主前置门禁。
- 今日相机网页端 `分类文件夹` 只保留为：
  - 登录 Chrome 当前可见会话下的备用人工归集入口
  - 后续 CLI 脚本归图与补差入口
  - 不能单独替代 SQL 主链完成口径
- 今日相机记录必须保留这些原始来源字段：
  - `sourceType`
  - `collectionSource`
  - `photoId`
  - `mediaUrl`
  - `takenAt`
  - `watermark`
- 今日相机路由规则当前固定为：
  - 显式带 `智店通入库群 / 教育补贴群` 的图片优先归对应群
  - 同手机号累计三件套默认归 `智店通入库群`
  - 普通单扫默认归 `教育补贴群`
- 当前首页 `教育补贴采集` 书签固定内嵌：
  - `bridge-dashboard.html`
  作为新采集主页面。

## 2026-06-11 新任务统一先拆解再路由，采用保守型模型分工

- 从现在起，联想智慧零售项目所有新任务统一执行：
  - `Codex 一级拆解 -> 按任务类别路由 -> 子智能体/模型执行窄任务 -> 回到 Codex 正式收口`
- 固定采用 `保守型` 方案：
  1. `Codex` 保留一级任务分解权
  2. `Codex` 保留正式业务收口权
  3. `GPT` 只做云端补料、公开信息整理、文本归纳，不写正式业务结论
  4. `OpenClaw` 只做副驾驶级窄任务，例如 receipt 级证据采集、巡检、阻塞上报、任务包执行，不再接管正式定时任务主链
- 四类任务固定分类：
  - `A类`：纯本地确定性任务，优先 `CLI`
  - `B类`：云端信息补料与归纳任务，优先 `GPT`
  - `C类`：仓库内工程实现任务，优先 `Codex`
  - `D类`：高风险真实页面任务，固定 `Codex` 主线程判定，`OpenClaw` 仅可做副驾驶
- 任何任务只要涉及以下任一项，最终都必须回到 `Codex`：
  - 改仓库文件
  - 改 SQL / SQLite / 正式快照
  - 判断页面真假、规格一致性、价格口径
  - 判定业务完成状态
  - 更新长期记忆
- 没有一级拆解卡片、没有执行单、没有产物路径的任务，不允许派发。
- 正式规则文档固定为：
  - `docs/ai-context/20_TASK_DECOMPOSITION_AND_MODEL_ROUTING.md`

## 2026-06-06 ERP/后台/台账页面禁止大方块卡片式占位，统一切换为紧凑式后台布局

- 从现在起，凡是命中以下页面类型：
  - `ERP`
  - `后台管理`
  - `订单管理`
  - `库存管理`
  - `SN 台账`
  - `出入库流水`
  - `价保汇总`
  - `实物仓管理`
  - 其它以 `台账 / 流水 / 审计 / 作业` 为主的页面
  统一禁止继续使用“大方块卡片式占位”作为主布局。
- 固定布局语言改为：
  1. `顶部工具条`
  2. `同层筛选栏`
  3. `紧凑 KPI 行或摘要行`
  4. `主台账表格`
  5. `作业区 / 审计区 / 明细区`
- 固定视觉约束：
  1. 表格和台账是一等公民，禁止再把核心数据拆成大量独立宫格卡片。
  2. 数量、金额、库存、比率统一右对齐并使用等宽数字。
  3. 状态用 `文字 + 标签色` 表达，禁止长段说明堆在大卡片里。
  4. 模块间采用细边框、浅底和紧凑 padding，避免大圆角、大阴影、大留白。
  5. 需要做模板页时，也必须按后台工具条 / 表格 / 台账的方式出样，不得再做营销卡片式比稿。
- 这条规则优先级高于个人审美偏好；后续只要是后台/ERP/库存/订单页面，默认视为硬规则。

## 2026-06-04 教育补代扫两个群统一采集标准，不再以箱码为主门槛

- 从现在起，`智店通入库群` 和 `教育补贴群` 的教育补代扫统一按同一套采集标准执行。
- 主采集证据固定为：
  - `智店通上报产品信息图`
  - `教育优惠券核销码图`
- 不再以箱码图作为主采集门槛；箱码图仅保留为辅助佐证。
- 统一归并规则：
  - `客户电话一致`
  - 或 `SN一致`
  - 或 `姓名一致`
  以上任一成立，即可认定为同一订单。
- 最小采集闭环固定为：
  - 普通单扫：`1` 个产品信息 + `1` 个核销码
  - 二件套：`2` 个产品信息 + `3` 个核销码
  - 三件套：`3` 个产品信息 + `4` 个核销码
- 固定回扫方向：
  - 先回到最新消息
  - 再从图片查看器按历史方向自底向上完整回扫到最早目标边界
  - 不允许只补最新、只补中段或只采局部
- 这条标准必须同步到：
  - 定时任务 prompt
  - 子智能体执行手册
  - 后续人工补录队列
  - SQL/快照持久化链
- 若只有箱码、没有产品信息图或核销码图，本轮只能记 `executed_not_closed`，不得写完成。

## 2026-06-04 UI/网页设计默认加载 `retail-web-ui + design-critic`，零售后台页面统一按表格/台账优先

- 从现在起，联想智慧零售项目里凡是命中以下场景：
  - `零售后台`
  - `门店管理`
  - `订单/库存/SN/客户/价格/促销`
  - `台账/流水/报表/对账`
  - `产品价保/教育补代扫/智店通镜像页`
  默认先按 `retail-web-ui` 技能执行，再用 `design-critic` 做版式和审美复核。
- 固定 UI 约束：
  1. 表格是一等公民；列表、流水、异常单、套装单优先做成台账式表格，不再堆文字块。
  2. 数字、金额、库存、数量、比率统一右对齐，开启等宽数字。
  3. 状态必须同时用 `文字 + 颜色 + 图标/标签` 表达，禁止只靠颜色或长文本堆叠。
  4. 间距统一采用 `4px` 基数；禁止继续出现 `5/7/9/13` 这类随意值。
  5. 中文后台默认正文 `14px`、模块标题 `20px`、卡片标题 `16px`、KPI `24-32px`。
  6. 单视图内语义色不超过 `3` 种，避免页面躁乱。
  7. 优先沿用当前 cockpit 已有交互与信息密度，不擅自发明脱离现有系统的新视觉语言。
- 固定验收约束：
  1. 改完前端后必须打开真实页面看实际布局，不接受只看代码或局部截图。
  2. 如果页面结构复杂或用户明确指出“难看/乱/堆叠”，默认再走一次 `design-critic` 式自查：
     - `Accessibility`
     - `Typography`
     - `Layout`
     - `Usability`
  3. 页面没打开、真实表格没看到、密度没复核前，不得声称 UI 已收口。

## 2026-05-26 定时任务不得私改主标题或真实销售价

- 从现在起，所有定时任务统一执行：
  - `采集证据 / 原始记录 -> SQL数据库 -> 审核后的发布投影 / snapshot_cache / API -> 前端只读展示`
- 禁止以下行为：

  - `product_master.canonical_name` 只允许人工产品库治理路径修改。
- 价格治理规则：
  - 外部采价统一是 `price signal`。
  - 门店零售价及其它销售相关最终价，只允许 `product_price_adjustment` 等显式 SQL 治理路径修改。

## 2026-05-24 书签页在线真值统一走 API -> SQLite，不允许继续把静态 JSON 当在线主源

- 从现在起，`apps/web-cockpit/src/App.tsx` 主书签页在正常在线访问场景下，真实数据链统一为：
  1. 前端 `service.ts`
  2. 本地 API
  3. SQLite 业务表或 SQLite `snapshot_cache`
- `public/data/latest-*.json` 只保留两种角色：
  - 离线静态包
  - SQL 镜像副本/证据副本
- 不再允许以下情况作为“在线已对接 SQL”的口径：
  - 前端虽然走 `/api/...`，但后端接口直接返回裸静态 JSON 文件
  - 前端在 `api_strict` 模式下默默回退到静态快照
  - 书签页渲染的是旧 `public/data`，却口头声称来自 SQL
- 当前允许的折中只有一种：
  - 上游仍由同步任务生成 JSON 快照，但 API 必须先把快照镜像到 SQLite `snapshot_cache`，再由书签页读取
- 后续新增书签页或新增子模块时，默认准入标准：
  - 能直接读 SQL 表则直接读 SQL 表
  - 还没建成规范表结构时，至少先纳入 SQLite `snapshot_cache`
  - 不允许新增“前端直读静态 JSON 作为在线主链”的实现
  - `api_strict` 模式下如果 API 失败，应返回空/失败态，而不是继续悄悄吃旧静态文件

## 2026-05-24 watchdog 只做提醒层，且必须随任务自动刷新

- `latest-scheduled-task-reports.json` 继续是业务完成状态唯一真值。
- `latest-scheduled-task-watchdog.json` 只负责：
  1. 新鲜度提醒
  2. 漏跑提示
  3. attention 汇总
- 从现在起，任何通过 `bash scripts/run_scheduled_task.sh <task>` 运行的任务，结束后都必须自动刷新 watchdog。
- `run-scheduled-task-watchdog` 返回 `2` 代表“有提醒项”，不是任务失败；包装脚本必须接受这个返回码，不能把它升级成执行失败。
- 对已统一迁移的新任务名，watchdog 允许保留旧键名回退兼容，但汇报口径仍统一使用新任务名。

  - `taste-skill` 只作为最后一层 UI 审核优化，不得单独主导 ERP/后台页结构
  - 没有真实本地页面验收，不得把 UI 写成已收口
