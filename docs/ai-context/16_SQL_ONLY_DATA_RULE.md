# SQL 主链硬规则（禁止 JSON 快照主存储）

更新时间：2026-05-29  
状态：强制生效（Hard Rule）

## 1. 目标

所有采集数据与同步数据必须以 SQL 表为唯一主链事实来源。  
JSON 仅允许作为前端静态缓存或导出产物，不得作为主存储、主对账依据或唯一恢复来源。

## 2. 强制范围

以下链路全部强制 SQL 主链：

1. 智店通销售出库 / 采购入库 / 其他出库 / 调拨
2. SN 台账、库存流水、订单流水
3. 教育补代扫、营销 PO、价保历史
4. 竞品、灰渠、分销报价（结构化后）
5. 客户资料、供应商资料、售后关联数据

## 3. 禁止项

1. 禁止“只写 JSON，不写 SQL”。
2. 禁止“先写 JSON，SQL 漏写也算完成”。
3. 禁止用 `latest-*.json` 作为唯一对账依据。
4. 禁止删除已入库 SQL 真实记录（只允许新增、状态变更、逻辑关闭）。
5. 禁止把“快照重建成功”当成“业务完成”。

## 4. 允许项（仅缓存层）

1. 可以从 SQL 生成 JSON 供前端展示加速。
2. JSON 必须标注 `source=sql.*` 与 `generatedAt`。
3. JSON 丢失时，必须可由 SQL 全量重建。

## 5. 执行门禁（任务完成前必须同时满足）

1. 采集数据已写入 SQL 目标表（有行数增量）。
2. SQL 对账通过（订单号/SN/金额/时间字段闭环）。
3. 前端读取链路已从 SQL 或 SQL 缓存镜像输出。
4. 若仅有 JSON 更新、SQL 无增量：任务状态只能是 `executed_not_closed`。

## 6. 最低验收清单

每次任务收口必须记录：

1. 写入表名
2. 写入条数（insert/update）
3. 去重键（如 `SN`、`order_number`、`movement_id`）
4. 缺口条数（sync_gap_queue open）
5. 前端可见数据校验结果

## 7. 违规判定

出现以下任一情况即判违规：

1. 前端显示有更新，但 SQL 对应表无新增/无变更。
2. 采集完成声明中未给出 SQL 写入证据。
3. 教育补代扫、出入库、订单只在 JSON 可见、SQL 不可查。

## 8. 既有任务整改要求

1. 所有定时任务提示词必须包含“SQL 优先、JSON 仅缓存”。
2. 现有使用 JSON 作为主链的任务，必须改为：
   - 先入 SQL
   - 再由 SQL 生成前端缓存
3. 旧逻辑未完成前，不得宣称“已真实完成”。
