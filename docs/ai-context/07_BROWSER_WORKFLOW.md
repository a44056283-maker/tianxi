## 远程域名访问验证

- 用户反馈远程域名、手机端、平板端白屏或数据加载慢时，不能只检查本机 `127.0.0.1:5174`。
- 固定检查顺序：
  1. `8000` 是否监听 `0.0.0.0:8000`
  2. 前端服务 `5174` 是否监听 `0.0.0.0:5174`
  3. 同源 `/api/...` 是否能通过 `5174` 代理到 `8000`
  4. 同源 `/data/*.json` 静态快照是否能直接返回
  5. 浏览器页面是否显示真实业务数据，而不是只显示壳
- 如果远程域名根路径或资源直接返回 `403`，优先检查 Vite `allowedHosts` 或正式网关 Host 配置；不要误判为业务数据不存在。
- 远程域名下禁止把数据源硬写为 `当前域名:8000`；标准做法是同源 `/api` + `/data` 静态快照兜底。
- 如果正式域名由 Nginx、Cloudflare Tunnel、Tailscale Funnel 或其它网关提供，网关必须把 `/api` 反向代理到本机 `8000`，把其余路径交给前端 `5174` 或静态站点。

## 本地网页微信 `https://localhost:3001/` 放行规则

- 本地网页微信固定目标是：`https://localhost:3001/`
- 如果 Chrome 出现：
  - `隐私设置错误`
  - `NET::ERR_CERT_AUTHORITY_INVALID`
  - `Connection Terminated`
  则先判定为本地浏览器/证书放行问题，不要直接误记为群聊或公众号本身异常。
- 当前可接受的真实完成口径只有两种：
  1. Chrome 活动标签页标题已进入 `WeChat Selkies` 或实际网页微信业务页
  2. 当前活动页 URL 仍为 `https://localhost:3001/`
- 如果只是 `curl -k` 可访问，但 Chrome 仍停在证书拦截页，不能写“已修复”。
- 当前项目已确认可用的浏览器侧修复口径是：
  - 先修容器证书为 `localhost` 证书链
  - 再让 Chrome 对 `localhost` 放行
- 当前不要把“系统所有客户端都已信任”写成完成标准；业务要求只需要 Chrome 当前稳定会话可访问即可。

## 全采集统一动作规则

- 本文件不是只管浏览器，而是所有“可见界面采集”的统一动作规范。
- 适用范围：
  - 微信
  - 公众号
  - 智店通
  - 京东
  - 联想官网
  - 天猫 / 淘宝
  - 任何需要人工可见操作的桌面程序或网页
- 固定动作顺序：
  1. 先扫描当前状态
  2. 再判断是否可操作
  3. 只执行一次手工单步动作
  4. 动作后再次扫描状态
  5. 再决定下一步
- “当前状态”固定要检查：
  - 当前窗口/标签页是不是目标
  - 当前是不是已登录
  - 当前是不是已加载完成
  - 当前有没有白屏、转圈、验证码、403、安全验证、错误弹层、菜单栏收缩、非目标页
  - 当前目标控件是不是稳定可见
- 明确禁止：
  - 连点
  - 双击补救
  - 未扫状态就点
  - 在滚动、重绘、转圈、错误状态下盲点
- 统一阻塞口径：
  - 只要扫描结果不是“目标页稳定可操作”，就必须立刻停下并记录阻塞

## 临时标签页清理规则

- 所有京东、联想官旗、天猫 / 淘宝、竞品排行和智店通采集任务，采集过程中允许打开临时商品详情页。
- 每个定时任务在每个站点只允许保留一个稳定工作页：
  - 网页微信固定为 `https://localhost:3001/`
  - 智店通固定为 `https://retail-pos.lenovo.com/`
  - 京东固定为当前唯一工作页
  - 联想官网固定为当前唯一工作页
  - 联想保修固定为当前唯一查询页
- 每个商品详情页完成核价、截图或证据记录并写入手工批次后，必须关闭该临时详情标签页。
- 每轮采集结束前必须清理本轮新开的临时标签页，只保留稳定入口页、登录页和必要任务页。
- 禁止把几十个商品详情页长期留在 Chrome 中；这会造成缓存堆积、内存占用和后续采集混乱。
- 同一站点不得并开多个业务标签页轮流采；如果已经积累出多个旧页，先手工收敛到唯一保留页，再继续本轮任务。
- 如果因为安全验证、403、验证码或页面卡死无法继续，先记录阻塞证据，再关闭本轮新开的无效临时页。

## 智店通

### 固定原则

- 所有手动操作先扫状态，再动作。
- 动作后必须再次扫状态，再决定下一步。
- 不允许把“连续点击”当成页面推进手段。
- 厂家系统疑似只允许同一浏览器保持一个稳定业务会话
- 不要同时开多个智店通业务标签
- 默认不走会话抓取和账号自动化作为日常主链路
- 库存同步默认遵守：
  - 先全量总表
  - 再页面补洞
  - 最后统一回写

### 智店通转圈/登录恢复固定动作

1. 如果目标业务页持续转圈、白页或卡死，不在原页硬等，也不连点。
2. 先等待最短必要时间，确认不是瞬时加载。
3. 刷新当前页一次。
4. 刷新后仍转圈或白页时，返回上一级功能入口后重新进入该页面。
5. 重新进入后必须再次选择当天开始日期、当天结束日期，并重新点击查询。
6. 只有完成“等待确认 -> 刷新一次 -> 返回上一级重进 -> 重新筛当天日期查询”后仍无法进入业务页，才允许进入登录恢复链。
7. 登录恢复链固定为：先关闭当前异常目标页，回到智店通首页或重新打开登录首页。
8. 点 `登录`：
   - 如果已保留会话且可自动进入，直接进入业务页。
   - 如果要求输入手机号，只输入 `15637798222` 后进入下一步。
   - 如果浏览器弹出已保存密码候选，只点已有候选密码登录，不手写、不保存新密码。
9. 如果出现短信验证码、二次认证、滑块或安全验证：
   - 立即停止当前路径
   - 任务状态写 `blocked_page_risk`
   - 通过飞书群提醒用户手动登录/验证
10. 如果重新进入后仍转圈或仍白页，同样写 `blocked_page_risk`，不得继续盲点。

### 库存同步执行顺序

1. 先在 `库存流水` 页面设定日期范围：
   - 起始固定为 `2025-10-01`
   - 结束为当日
2. 再分别选择类型后导出：
   - `入库`
   - `出库`
3. 两份导出先进入本地总表事实层：
   - 用来对齐 `SKU / 单号 / 进货时间 / 出库时间 / 操作人 / 可回放流水`
4. 再用：
   - `商品库存有库存导出`
   - `商品库存SN导出`
   - `SN库存订单`
   去补当前在库真值和序列号层
5. 只有当总表里出现明确缺口时，才允许进入页面单采补洞：
   - `商品库存 -> 查看序列号`
   - `SN库存订单`
6. 页面补洞时只处理：
   - `currentStock > serialCount`
   - 单据字段缺失
   - 操作人缺失
   - 供应商链缺失
7. 页面补洞不能边采边同步：
   - 先归集成手工批次 JSON
   - 再统一执行 `save:serial-overrides` 或对应导入器
   - 最后统一重建快照并刷新前端
8. 禁止把页面逐条补录重新当成主流程。

### 入库采集

路径：

`库存 -> 出入库 -> 商品入库`

规则：

- 默认只采当天
- 记录：
  - 业务单号
  - 入库时间
  - 操作人
  - 供应商
  - 门店
  - 库位
  - SKU / PN / 规格
  - SN 列表
- 保存为本地 JSON 后导入

### 销售出库采集

路径：

`订单 -> 线下门店订单 -> 下单/完成时间筛选 -> 已完成`

规则：

- 当前日常口径：采集昨天和今天
- 后续日常可能收紧为“当天增量 + 必要补采”

### 其他出库采集

路径：

`库存 -> 出入库 -> 其他出入库`

规则：

- 只用于库存校准
- 不并入零售销售统计
- 只采当天
- 如果存在重复业务单，按用户明确口径排除；当前已知排除项：
  - `KCM26051307718`
- 打开业务单后不能只记录数量摘要，必须继续进入明细层采集：
  - `SKU`
  - `PN`
  - `规格`
  - `数量`
  - `SN 列表`
- 如果当天只采到了数量、成本、金额，没有逐台 `SN`，则只能算：
  - `数量型其它出库已同步`
  - 不能算 `SN 级其它出库已同步`

### 子代理使用方式

- 子代理可先做：
  - 报告核查
  - 原始文件是否到位核查
  - 缺口统计
- 真实页面采集仍由主线程执行。

## 京东网页

### 当前规则

- 先搜自营店
- 自营没有，再搜京东全站
- 搜索顺序固定为 `型号` -> `型号 + 核心配置` -> `型号 + 核心配置 + 颜色`
- 首轮不允许直接整串完整配置硬搜到底
- 已锁定详情页优先
- 只处理有库存 SKU
- 用户提供京东详情页时，默认用 Chrome 打开核验，不再改用其他浏览器。
- 已实际打开核验后才能写“已读取/已看到”；未打开不得用含混说法替代核验。
- 用户提供详情页且型号匹配时，按快速链路执行：核验标题/配置/店铺/价格 -> 写入快照 -> 重建前端数据 -> 从缺价队列移除。
- 系列页商品必须继续点到正确的处理器、颜色、版本、存储子配置；系列对但子配置错，不能写回。
- 京东指定店铺详情页失效时，不停在 `仅入口`：
  - 必须立即转 `京东全站` 补搜新的真实详情页
  - 补到新详情页后再更新链接和价格
- 联想官旗详情页下架或失效时：
  - 官旗旧链接和旧价格立即失效
  - 必须转 `天猫 / 淘宝` 的已确认在售页补替换链接与价格

### 禁止动作

- 在详情页做系列切换
- 高频来回刷新
- 反复打开已采过详情页
- 多型号混挂页强行点配置
- 立即购买、复杂营销弹层等高风险点击

### 价格规则

- 主价格取国补前展示整数价
- 京东页面直接可见满减/平台券金额时，主价格采信券后国补前价
- 页面原价、满减/平台券、教育补、国家补贴都要单独记录
- 教育补、国家补贴不直接覆盖主价格
- 带小数价格通常视为补贴后价或活动价线索
- 如果 JD 页面只有最终到手价，且叠加了首发满减、学生券、教育补、PLUS 立减等多层活动，则当前轮主零售价先与匹配官旗零售价同步；JD 页面的最终到手价只保留为活动线索，后续再同步到营销英雄卡
- 京东、联想官网、天猫/淘宝、网页微信、智店通任一站点如果出现登录失效、403、验证码、短信验证、滑块或安全验证：
  - 立即停止当前路径
  - 不换新浏览器、不脚本重试、不高频刷新
  - 通过飞书群播报“需要重新登录/验证”

### 平台差异规则

- `联想官网商城`：
  - 秒杀价视为正常卖价
  - 不做 `99` 元收尾
  - 不把秒杀价反推成所谓补贴前价
- `京东第三方店`：
  - 如果页面没有国补口径，采优惠前价作为补贴前采集价
  - 最终匹配价统一按 `99` 元收尾
  - 例如优惠前价 `6985`，最终按规则收尾为 `6999`
  - 如果页面明确展示国家补贴金额和国补前整数价，则直接采该国补前整数价
  - 例如红字价 `3909.15`、国补前价 `4599`、国家补贴 `689.85`，最终主价采 `4599`
- `京东自营/明确国补场景`：
  - 仍按页面可见主价、券后国补前价、用户补充规则综合采信
  - `PLUS`、返京豆、教育补等说明项单独保留，不直接覆盖主价

## 微信 / 公众号

### 总规则

- 微信报价采集固定使用 Chrome 中已经登录的 `https://localhost:3001/` 网页微信，不打开微信桌面版。
- 微信属于高风控人工场景。
- 所有动作必须遵循：
  1. 先看当前网页状态
  2. 确认目标页稳定存在
  3. 执行一次单步动作
  4. 再重新看状态
- 不允许：
  - 打开微信桌面版作为报价采集入口
  - 连点
  - 双击文章卡片
  - 盲坐标点击正文区
  - 在登录页、二维码页、底部菜单栏收缩态继续硬操作

### 遇到以下状态立即停止

- 扫码登录
- 仅传输文件
- 公众号白屏
- 公众号正文未展开但窗口缩到底部菜单栏
- 非目标公众号/非目标聊天
- 风控提示页

### 采集目标

- `https://localhost:3001/` 网页微信只负责把当天目标内容稳定显示出来
- 原文保存、截图、OCR、结构化解析由后续步骤接手
- 不允许为了追求速度牺牲登录态和页面稳定态
- 分销群报价必须先在网页微信可见界面确认群聊、文件入口或当天报价文件；如果当天 Excel 已经由 Selkies 映射目录落盘，允许直接消费该当天文件，但必须在报告里写清它来自网页微信映射目录。
- 灰渠公众号必须从网页微信 `文件传输助手` 聊天记录区进入，只使用文件传输助手聊天记录下面用户固定放置的公众号入口；进入公众号页只算入口到达，不算采集完成。
- 进入公众号后必须点击页面最下面带日期的报价快捷入口，日期必须为当天或当前最新有效报价日期；只认可该快捷入口打开后的报价页、当天截图或 OCR 证据，再运行本地解析和 SQL/API/前端同步。
- 禁止退回收藏夹、公众号文章列表、搜索公众号、桌面微信、旧原文或旧截图重跑来替代最下面带日期的报价快捷入口。
- 若执行记录仍指向历史错误入口，按旧流程残留处理，先修正规则再采集。
