# 22 Standard Task Cards And Dispatch Table

更新时间：2026-06-11

## 目标

把“先拆解、再路由、再派发、再收口”进一步固定成可直接复用的表单。

以后新任务默认按以下顺序执行：

1. `Codex` 填一级拆解卡片
2. `Codex` 判断任务类别与风险
3. 选择执行者：`CLI / Codex / GPT / OpenClaw`
4. 生成对应执行单
5. 执行完成后回到 `Codex` 收口

本文件配套：

- `docs/ai-context/20_TASK_DECOMPOSITION_AND_MODEL_ROUTING.md`
- `docs/ai-context/21_DISPATCH_ENTRYPOINTS_AND_MANUAL_HANDOFF.md`

## 二、三模型派发表

| 任务场景 | 默认类别 | 默认执行者 | 是否允许直接派发 | Codex 是否必须收口 |
|---|---|---|---|---|
| 快照重建、报表重算、上下文打包 | A | CLI | 是 | 是 |
| 公开资料补料、长文归纳、规则初稿 | B | GPT | 是 | 是 |

| 只读对比、计划表整理、待办归纳 | A / B | GPT 或 OpenClaw | 是 | 是 |

## 三、执行者选择口诀

### 1. 先问四个问题

1. 要不要改仓库文件？
2. 要不要判断真实页面真假或规格一致性？
3. 要不要写正式快照、SQL、SQLite？
4. 产出是不是只读草稿、清单、摘要？

### 2. 决策规则

- 只要第 `1/2/3` 任一为“是”，默认回到 `Codex`
- 只有第 `4` 为“是”且不碰正式写入时，才优先考虑 `GPT`
- 只要任务属于：
  - receipt 回传
  - 状态巡检
  - 阻塞上报
  - board 审计
  就优先考虑 `OpenClaw`
- 只要任务是纯本地确定性重建，就优先交给 `CLI`

## 五、任务分派红线

以下任务禁止直接派给 `GPT`：

- 京东 / 联想官网 / 天猫真实价格判定
- 智店通登录态页面采集
- 微信群当天原文采集
- 正式快照写回
- SQL / SQLite 主表写入

以下任务禁止直接整包派给 `OpenClaw`：

- 正式定时任务主链
- 正式业务完成判定
- 无人工边界的真实页面长流程
- 任何需要直接宣称“已同步完成”的任务

以下任务不该占用 `Codex` 主线程：

- 机械式日报拼装
- 重复性状态抄写
- 可由脚本直接完成的快照重建
- 只读型 receipt 汇总

## 六、首次推荐使用方式

以后用户只要说：

- “继续吧”
- “开始做”
- “把这个任务分给别的智能体”
- “先拆一下”

主线程默认执行：

1. 先写 `一级拆解卡`
2. 再选执行者
3. 再生成执行单
4. 再投递或执行
5. 最后由 `Codex` 汇报“已完成 / 未完成 / 阻塞”

## 七、当前项目推荐默认档位

- `CLI`
  - 快照重建
  - 审计报表重算
  - context 打包
- `Codex`
  - 仓库改动
  - 验证
  - 长期记忆更新
  - 正式收口
- `GPT`
  - 云端补料
  - 文档归纳
  - 公开资料候选整理
- `OpenClaw`
  - receipt 级只读任务
  - 状态巡检
  - 阻塞上报
  - command/chat board 协作任务

## 八、当前已验证的最小闭环

截至 `2026-06-11`，当前已验证的不是“三模型全自动编排”，而是以下保守闭环：

1. `Codex` 先拆解
2. `Codex` 手动生成 `OpenClaw` 执行单
3. `OpenClaw` 通过本机网关真实接收任务
4. `OpenClaw` 回传 chat feedback 或隔离 receipt
5. `Codex` 基于 receipt / board 事实做最终收口

`GPT` 当前仍是人工投递链，不是仓库内自动派发链。
