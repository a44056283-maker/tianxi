# 20 Task Decomposition And Model Routing

更新时间：2026-06-11

## 目标

从现在起，本项目所有新任务默认先拆解，再按模型能力路由，再执行，再回到 Codex 收口。

本规则采用用户确认的 `保守型` 方案：

- `Codex` 保留一级任务分解权
- `Codex` 保留正式业务收口权
- `GPT` 与 `OpenClaw` 只作为副驾驶，不得越权写成“业务已完成”

这份规则适用于：

- 日常开发任务
- 定时任务整改
- 数据同步任务
- 外部资料补料
- 页面证据采集
- 子代理并行任务

不适用于：

- 用户明确要求只做单点只读查询
- 用户明确要求跳过分发、只让 Codex 单线程完成的小修小补

配套落地模板见：

- `docs/ai-context/22_STANDARD_TASK_CARDS_AND_DISPATCH_TABLE.md`

## 一、总原则

1. 所有新任务先由 `Codex` 做一级拆解，禁止直接把原始大任务丢给 `GPT` 或 `OpenClaw`。
2. 一级拆解必须先判断：
   - 是否涉及真实页面
   - 是否涉及正式数据写入
   - 是否涉及 SQL / API / 前端三层闭环
   - 是否允许并行
   - 最终由谁收口
3. 任何涉及正式业务真值、正式快照、正式 SQL、正式前端展示的任务，最终都必须回到 `Codex` 收口。
4. `GPT` 与 `OpenClaw` 的输出默认只算“中间产物”，不算“主链完成”。
5. 没有拆解卡片、没有执行单、没有产物路径的任务，不允许派发。

## 二、四类任务分类

### A 类：纯本地确定性任务

特征：

- 不需要云端补料
- 不需要真实页面判断
- 不需要人工解释复杂业务口径
- 输入和输出都在本仓库或本机可控范围内

典型任务：

- 快照重建
- SQL 镜像同步
- 审计报表生成
- 上下文打包
- 日报拼装
- 备份
- receipt / command board 汇总

默认路由：

- 优先 `CLI`
- 需要人调度时由 `Codex` 或 `OpenClaw` 触发
- 最终状态仍由 `Codex` 审核

### B 类：云端信息补料与文本归纳任务

特征：

- 允许使用云端模型能力
- 不直接写本地正式业务真值
- 产出主要是草稿、清单、摘要、候选、归纳

典型任务：

- 公开资料汇总
- 规则文档初稿整理
- 非登录态网页信息摘要
- 长对话 / 长文档归纳
- 竞品公开信息初筛

默认路由：

- 优先 `GPT`
- 输出必须回交 `Codex`
- 不能直接改正式业务快照

### C 类：仓库内工程实现任务

特征：

- 需要修改仓库代码、脚本、配置、文档
- 需要本地验证
- 需要理解现有工程结构和业务主链

典型任务：

- 前端修复
- API/SQL 链路修复
- CLI 改造
- 自动化脚本整改
- 规则文档落地

默认路由：

- 优先 `Codex`
- 可以把只读审计子任务并行分给其它智能体
- 正式改动、验证和记忆更新由 `Codex` 完成

### D 类：高风险真实页面任务

特征：

- 涉及智店通、京东、联想官旗、网页微信等真实页面
- 依赖登录态、规格点击、页面判读、风控识别
- 不能靠旧文件重跑假装完成

典型任务：

- 京东/联想官旗价格复核
- 智店通库存/SN/出入库页面核查
- 微信分销群报价/灰渠当天原文取证
- 保修页人工核对

默认路由：

- 主线程 `Codex` 负责判定与收口
- `OpenClaw` 只允许做副驾驶级证据采集、巡检、receipt 回传、阻塞上报
- 不允许把这类任务整包交给 `GPT`

## 三、三种执行角色

### 1. Codex

固定职责：

- 一级任务分解
- 风险判断
- 文件改动
- 正式导入与正式写入
- SQL / API / 前端三层验收
- 长期记忆更新
- 最终汇报

禁止降级去做的工作：

- 大批量低价值日报拼接
- 重复性纯报表搬运
- 机械式状态抄写
- 可由脚本直接完成的确定性重建

### 2. GPT

固定职责：

- 云端资料补料
- 文本归纳
- 公开信息候选整理
- 规则草稿、提纲、摘要生成

禁止事项：

- 不直接改仓库代码
- 不直接写正式业务快照
- 不直接判定真实页面采集结果
- 不写“已完成同步”

### 3. OpenClaw

固定职责：

- 接收 `Codex` 手动下发的窄任务
- 做 receipt 级证据采集
- 做状态巡检
- 做阻塞上报
- 做任务包执行结果回传

禁止事项：

- 不接管正式定时任务主链
- 不直接写正式 `latest-*.json`
- 不直接写 SQLite / SQL 主业务表
- 不直接宣称“业务已完成”
- 不绕过页面风控、验证码、403、白屏

## 四、一级拆解卡片必填项

以后 `Codex` 收到新任务后，必须先在工作过程里明确以下字段：

```text
任务名称：
任务目标：
本轮只做什么：
本轮不做什么：
任务类别：A / B / C / D
风险级别：低 / 中 / 高
是否涉及真实页面：是 / 否
是否涉及正式写入：是 / 否
是否允许并行：是 / 否
建议执行者：Codex / GPT / OpenClaw / CLI
最终收口者：Codex
预期产物：
验证方式：
```

没有这张卡片，不允许往下派发。

## 五、模型路由规则

### 固定路由

1. `A 类`
   - 默认 `CLI`
   - `Codex` 负责触发、审计和汇报
2. `B 类`
   - 默认 `GPT`
   - 只回传候选、摘要、草稿、结构化结果
3. `C 类`
   - 默认 `Codex`
   - 如有并行只读支线，可拆给其它智能体
4. `D 类`
   - 默认 `Codex + OpenClaw 副驾驶`
   - 不允许只交给 `GPT`

### 强制回退到 Codex 的条件

出现以下任一情况，任务必须回到 `Codex`：

1. 需要改仓库文件
2. 需要改 SQL / SQLite / 正式快照
3. 需要判断页面真假、规格一致性、价格口径
4. 需要执行最终业务状态判定
5. 需要更新长期记忆
6. 出现阻塞、冲突、证据不完整或模型输出自相矛盾

## 七、手动分发 SOP

### 步骤 1：Codex 先拆主任务

先写一级拆解卡片，再决定是否分发。

### 步骤 2：判定是否允许外发

满足以下全部条件才允许外发：

1. 任务边界窄
2. 输入来源清楚
3. 输出格式清楚
4. 不要求对正式业务完成做最终判定
5. 不要求直接改主仓正式真值

### 步骤 3：生成执行单

根据目标智能体生成：

- `Codex 执行单`
- `GPT 执行单`
- `OpenClaw 执行单`

### 步骤 4：下发

1. `GPT`
   - 只下发补料、归纳、摘要型任务
2. `OpenClaw`
   - 只下发 receipt / 巡检 / 证据采集 / 阻塞上报型任务

### 步骤 5：回收产物

回收时必须检查：

1. 有没有新产物
2. 产物路径是否真实存在
3. 输出是否越权
4. 是否需要回到 Codex 继续处理

### 步骤 6：Codex 最终收口

只有 `Codex` 可以把状态写成以下之一：

- `real_completed`
- `executed_not_closed`
- `blocked_missing_input`
- `blocked_page_risk`

`GPT` 与 `OpenClaw` 只能提交中间结果，不能宣判最终状态。

## 八、产物门禁

以后任何被分发出去的任务，至少要新增以下产物之一：

1. 新的结构化草稿或候选清单
2. 新的 receipt / command / 证据文件
3. 新的阻塞报告
4. 新的代码改动、脚本改动或文档改动

如果没有新增产物，只能写：

```text
execution_failed_noop
```

## 九、OpenClaw 在本项目内的保守型定位

从现在起统一口径如下：

1. `OpenClaw` 可以被 `Codex` 手动下发窄任务。
2. 这些任务仅限：
   - receipt 级证据采集
   - 巡检
   - 阻塞上报
   - 任务包执行
3. `OpenClaw` 不是正式定时任务主执行器。
4. `OpenClaw` 不是正式 SQL / 快照 / 前端同步收口者。
5. 只要结果要写入正式业务链，最后一跳必须回到 `Codex`。

除现有固定记忆入口外，凡是命中多任务、模型分工、定时任务改造、OpenClaw/GPT 分发、子代理并行，额外读取：

- `docs/ai-context/20_TASK_DECOMPOSITION_AND_MODEL_ROUTING.md`

## 十一、示例

### 示例 1：让 GPT 处理

任务：

- 汇总公开竞品资料并整理成候选对比表

路由：

- `B 类 -> GPT`

Codex 收口：

- 复核后再决定是否写进项目文档

### 示例 2：让 OpenClaw 处理

任务：

- 巡检某个已定义 receipt 任务是否产生新证据并上报阻塞

路由：

- `A/D 之间的副驾驶子任务 -> OpenClaw`

Codex 收口：

- 读取 receipt，决定是否继续导入或判阻塞

### 示例 3：必须 Codex 自己做

任务：

- 修复零售快照计算逻辑并重建前端数据

路由：

- `C 类 -> Codex`

原因：

- 涉及代码改动、正式写入和验证，不能外包
