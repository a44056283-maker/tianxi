  1. 已真实完成
  2. 已进入明确阻塞，且当前没有进一步可执行解法
  3. 继续操作会带来高风险破坏
- 禁止：
  - 问“下一步要不要继续”
  - 在还能继续修的时候先做阶段性收口
  - 把“准备做”当成对用户的交付

## 3.3 多任务并行与模型分级

- 只要用户下达的是多任务，不允许再按单线程顺序慢慢做；必须先拆任务，再并行分派。
- 任务拆解与模型路由的正式执行细则，统一以：
  - `docs/ai-context/20_TASK_DECOMPOSITION_AND_MODEL_ROUTING.md`
  为准。
- 主线程固定先做：
  1. 识别关键路径
  2. 决定哪些结果会阻塞下一步
  3. 把不阻塞主线的支线任务分给子智能体并行执行
- 子智能体分派必须写清：
  1. 本轮只做什么
  2. 本轮不做什么
  3. 输入文件或输入来源
  4. 成功标准
  5. 输出路径或汇报格式
- 默认模型分级：
  - `gpt-5.5`：只用于高风险、跨模块、强推理、真实业务口径容易误判的复杂任务；默认不进入日常定时任务。
  - `gpt-5.4`：用于智店通、京东/联想价格复核、竞品详情核验、网页微信/灰渠等需要真实页面判断的中高复杂任务。
  - `gpt-5.4-mini`：用于日报/播报、看门狗、快照重建、开发计划整理、陈旧库存巡检、纯报表审计等低风险任务。
- 当前项目硬规则：
  - 日常定时任务默认不使用 `gpt-5.5`
  - 只有在“新规则刚上线、任务连环误判、前端/API/SQL 三层互相打架、或真实页面证据与本地结果明显冲突”时，才允许把单个任务临时升到 `gpt-5.5`
  - 任何降级都不能把真实页面采集任务降到无法守住业务边界的程度
- 汇报必须区分：
  - “并行只读审计”
  - “并行规则整理”
  - “主线程真实执行/真实写入”
  不能把子智能体只读核查写成主线已同步完成

## 3.3.1 定时任务执行力门禁

- 定时任务下发以后，默认目标是实际执行，不是只读审计。
- 每轮任务必须至少新增以下产物之一：
  1. 新的正式任务报告
  2. 新的手工批次或证据文件
  3. 新的阻塞证据文件，并写明无法继续的具体原因
- 如果三类产物都没有，本轮必须判定为 `execution_failed_noop`。
- 禁止把以下行为写成“已执行”：
  - 只读报告
  - 只看 `git status`
  - 只写 memory
  - 只做局部取证
  - 只发送飞书播报但不核对数据源和前端显示
- 价格、链接、竞品、国补类任务遇到页面存在颜色、配置、套餐、版本、规格区或页面下方同系列规格时，必须点击或切换到目标 SKU 对应规格，并记录选中规格文案、标题、配置、主价/国补前价和链接证据。
- 未完成规格点击与选中态记录时，禁止判定“无同配”，禁止写回价格或链接，只能保留为待复核或页面阻塞。
- 价格、链接、竞品、国补类任务的标题证据必须结构化保存 `平台主标题 / 小字副标题或配置副标题 / 已选规格文案 / 商品编号`。只采主标题、不采副标题或已选规格，按证据不完整处理。
- 价格字段必须结构化拆成 `preSubsidyPrice / couponAdjustedPrice / postSubsidyPrice`。门店主参考价只能取营销、教育、PO、国补前原始正常价，不能取券后中间价或国补后红字到手价。
- 平台详情页若当前目标规格显示 `无货 / 到货通知 / 售罄 / 暂不销售 / 待发布`，该来源只保留证据和链接，不得继续参与门店主参考价；主参考价必须降级切到另一个“有价且可售”的来源。
- 白色、黑色、灰色等颜色差异属于同配判定一部分。游戏本白色款不得直接沿用黑色价格；只有目标颜色被真实点选且标题、规格、价格同页一致时，才允许写回。
- 超过 `36` 小时未复核的平台价，只能保留历史证据和固定链接，不得继续参与当前主价格计算。
- 只要 SKU 或页面存在白色/冰魄白/月幕白等白色变体，就必须额外执行一次白色复采核对：再次确认白色规格的标题、副标题、已选规格文案、商品编号和主价口径；没有完成这次白色复采核对，任务只能写 `executed_not_closed`。
- 涉及网页采集的任务必须先收敛站点标签页：每个站点每轮只保留一个稳定工作页，临时详情页或候选页用完立即关闭，不能把多标签并行浏览当作默认采集方式。
- 涉及网页采集的任务必须先做标签页数量前置检查：记录收敛前页数、收敛后页数，再进入真实采集。没有前置页数记录，不能把任务写成已真实完成。
- 所有定时任务结束时必须把 Chrome 归位到固定 `5` 页恢复态；不能因为“本轮只是少量采集”或“本轮只是播报/审计”就跳过归位。
- 固定 `5` 页恢复态口径：`WeChat Selkies / https://localhost:3001/`、`店铺列表 - 智慧零售云平台 / https://retail-pos.lenovo.com/`、`联想智慧零售系统 / http://127.0.0.1:5174/`、`联想官网`、`京东`。
- 对播报和快照重建类任务，脚本成功只代表本地命令成功；必须核对数据源和前端对应子书签。没有前端可见审计时，只能写 `executed_not_closed`。
- 任何智能体不得私自退出、停用、暂停、删除、覆盖、降级或改为非 `ACTIVE` 的既有定时任务；未完成时只能如实保留任务并反馈阻塞。
- 若任务未完成、自动化列表掉线、任务状态异常或被外部操作挪动，必须立即向飞书群反馈，不得静默结束。
- 定时任务恢复、重启或重新加载后，禁止自动批量补跑历史未完成任务。自动执行只处理当前时间窗到期的当前轮，历史漏跑/阻塞只保留证据和提醒，不进入自动排队。
- 同一计划时段只允许自动尝试一次；不能在同一窗口内反复重试，更不能把旧时段和新时段任务堆在一起执行。
- 任何智能体采集到的数据任务，只有在 `SQL 持久化 -> API/受控快照刷新 -> 前端 UI 亲眼验收` 三层全部闭合后，才允许写 `real_completed`。
- 只完成采集、只完成本地 JSON、只写日志、只写 SQL 但未刷新前端、或前端 UI 仍是旧值，都只能写 `executed_not_closed`。
- 汇报必须同时给出 SQL 写入结果、前端 URL、子书签、可见关键字段与验收时间；没有 UI 实证，不得把任务写成完成。

## 3.0 远程域名访问执行规则

- 后续所有前端、API、定时任务看板、价保、库存、报价、产品库模块开发，都必须按“本机 + 局域网 + 远程域名”三种访问方式设计。
- 本机可以直接访问 `127.0.0.1:8000`；远程域名默认只能假设访问同源 `/api` 和同源 `/data`，不能假设浏览器能直接访问裸露的 `:8000` 端口。
- 页面读取数据必须采用分布式加载：
  - 首屏先保留骨架/旧数据，不因单个接口失败导致全页白屏
  - 独立模块独立加载，失败模块降级为静态快照或明确阻塞
  - API 慢或失败时优先使用 `/data/*.json` 快照兜底
- 新增读取函数必须具备静态快照 fallback；新增写入函数可以只支持 API，但要在远程只读模式下明确不可编辑。
- 验证不能只跑 `127.0.0.1:5174`；涉及展示问题时至少同时检查：
  - `127.0.0.1:5174/api/...`
  - `127.0.0.1:5174/data/...`
  - 局域网 IP 或远程域名下的同源 `/api` 与 `/data` 是否返回。

## 3.1 全采集统一执行协议

- 外部登录态采集硬门禁：
  - 智店通、网页微信、京东、联想官旗、天猫、淘宝等外部页面只能使用用户当前已登录的默认 Chrome `user` 会话。
  - 禁止打开新的浏览器、空白浏览器、新 Chrome Profile，禁止清理登录缓存，禁止退出账号后重登。
  - 禁止使用 Browser、in-app browser、browser-use、Playwright、Puppeteer、Chromium launch 打开或推进外部采集页面。
  - 如果当前老 Chrome 会话未登录、出现二维码、验证码、白屏、安全验证或无法进入目标页，立即记录 `blocked_page_risk`，不得换浏览器绕过。
- 教育补代扫箱码硬门禁：
  - 智店通入库群必须从当前网页微信会话列表或右侧信息栏进入，禁止用搜索栏搜索群名后冒充已到位。
  - 必须历史方向上翻到上一次已确认的最后一张代扫教育补箱码图片或明确边界。
  - 进入群后打开看到的第一张相关照片，优先用图片查看器左箭头或键盘左键逐张向历史图片方向查看，直到目标图、对应箱码图、上传完成卡、核销成功卡和最后一次已经采集过的代扫教育补照片边界都确认。
  - 采集后必须同步写入 SQL 主链，刷新前端，并检查前端教育补/出入库流水同步可见；否则不能写 `real_completed`。
  - 未到箱码边界前，不能写 `confirmedNoNewRecords`，只能写 `executed_not_closed`。
  - 只看到代扫文字、上传完成卡或核销成功卡但没找到对应箱码，也只能写 `executed_not_closed`。
- 任何采集型任务，无论来源是微信、公众号、智店通、京东、联想官网、天猫、淘宝、桌面程序还是浏览器页面，都必须先执行同一套协议：
  1. 扫描目标现状
  2. 判定是否可操作
  3. 只执行一次单步动作
  4. 再次扫描状态
  5. 再决定下一步
- 这里的“目标现状”至少包含：
  - 当前是否在目标窗口/目标标签页
  - 当前是否已登录
  - 当前页面是否完成加载
  - 当前是否存在白屏、转圈、验证码、403、安全验证、错误弹层、菜单栏收缩、非目标页
  - 当前目标控件是否稳定可见、不是滚动中或重绘中
- 统一禁止：
  - 未扫描状态就直接点击
  - 用连点、双击、盲点补救
  - 在错误状态下继续推进
  - 把“还能点”当成“可以继续采”
- 统一阻塞规则：
  - 只要扫描结果不是“目标页稳定可操作”，任务就必须立刻停下，记录真实阻塞

- 自动化定义落盘到 `../automation_payloads.json` 和 `~/.codex/automations/*/automation.toml` 时，必须包含同一规则块，避免任务智能体只读局部提示词后漏掉通用采集规则。

## 3.2 全采集统一提示词模板

后续所有采集类提示词、教程、子代理任务单、自动化任务说明，都必须至少包含下面 5 句：

```text
1. 操作前先扫描目标当前状态，确认当前窗口、页面、登录态、加载态、目标控件是否正确。
2. 只有目标页稳定可操作时，才允许执行一次手工单步动作。
3. 每执行一步后，必须再次扫描状态，再决定下一步。
4. 出现登录页、二维码、白屏、转圈超时、403、安全验证、错误弹层、底部菜单栏收缩或非目标页，立即停止并写阻塞。
5. 禁止连点、双击、盲点、错误状态下硬操作。
```

如果某份教程、任务单或提示词缺少这 5 句，就视为规则不完整，不能进入正式执行。

   - 只凭页面少量记录覆盖全量总表
   - 把单采描述成“已完成完整同步”

## 标准报价总表执行规范

### 一、固定顺序

1. 原始采集记录先进入 `latest-marketplace-price-snapshot.json`
2. 再按 `SKU x 来源(jd / lenovo_official)` 归集到 `latest-standard-price-master.json`
3. 总表层完成比对和证据审计后，再派生 `latest-standard-price-master-frontend-snapshot.json`
4. 前端、计划面板、后续统一同步只读总表或前端快照，不直接消费零散原始记录

### 二、归集规则

- 总表只接受真实商品详情页作为正式来源：
  - 京东：`item.jd.com/<数字>.html`
  - 联想官旗：`item.lenovo.com.cn/product/<数字>.html`
- 搜索页、店铺页、活动页只允许保留在线索层，`evidenceAudit.status` 必须落成 `non_detail_url` 或 `placeholder_only`
- 同一 `SKU x 来源` 有多条原始记录时：
  - 优先详情页
  - 再优先 `captured + confirmed`
  - 再优先证据完整
  - 再优先最新采集时间

### 三、比对规则

- 只在京东和联想官旗都满足以下条件时进入正式比对：
  - 已锁定详情页 URL
  - 有可比较主报价字段
  - 证据字段完整
  - 原始状态不是 `manual_review_required / url_configured_only / unavailable`
- 比对结果分为：
  - `ready_for_compare`
  - `missing_source`
  - `manual_review_required`
  - `evidence_incomplete`
  - `unavailable`

### 四、统一同步规则

- 只有 `syncDecision.status = ready_for_unified_sync` 的 SKU 才能进入统一同步候选。
- 价差大于 100 元时，不自动统一，改为 `hold_large_gap_review`。
- 当前骨架的统一同步候选价规则：
  - 两端同价：直接取该价
  - 两端都合规但不一致：先生成较低展示价作为候选值
  - 任何一端证据不完整、未锁链、待复核：一律阻断

### 五、人工采集证据边界

人工或类人工采集记录必须同时具备：

- `evidenceUrl`
- `screenshotPath`
- `capturedAt`
- `capturedBy`
- `note`

其中：

- `note` 必须写清价格口径，至少说明是否券后、是否国补后、页面是否可销售。
- 没截图、没时间、没详情页 URL 的记录，只能留在原始层或总表待复核状态，不能冒充正式报价。
- 下架、无货、待发布页面可以保留证据，但不能进入统一同步。
## 智店通单笔手工出库补录规则

1. `save-inventory-movements` 是整包覆盖语义，不是 append。
2. 手工补录单笔销售/入库/其他出库时，必须先读取当前全量 `latest-inventory-movements.json.records`。
3. 只把新增记录按 `id` 合并进全量数组后，再整包写回。
4. 禁止直接把单条补录 JSON 传给 `save-inventory-movements`，否则会把主流水覆盖成单条快照。
5. 整包写回后固定重建：
   - `node --import tsx/esm src/cli.ts build-inventory-master`
   - `npm run build:marketing-boost`
   - `npm run build:retail-zone`
   - `retail_core.seed_reference_data(...)`
