## 总原则

1. 每日定时任务的含义是“每天必须做完的工作”，不是“到点触发过脚本”。
2. 任务状态只能分为：
   - `已真实完成`
   - `已执行但未收口`
   - `因缺原始数据阻塞`
   - `因页面/风控阻塞`
3. 手工采集任务必须保留：
   - 采集时间
   - 采集人/执行人
   - 原始文件、截图或页面证据
   - 写回的目标快照
4. 已存在的零售详情页链接优先直接打开核价，不重复搜索。
5. 京东/联想官旗采价遵循已有项目价格规则，不允许现场临时发挥。
6. 库存完整信息总表必须走“人工导出 -> 本地合并 -> 同步快照”的链路，不允许跳过原始导出直接手填主表。
7. 智店通库存主链固定三步：
   - 先同步当天采购入库、销售出库、其他出库
   - 再导出当天完整 `商品库存统计` 与 `商品库存SN统计`
   - 最后只用这对成对总表统一刷新前端库存与 SN
8. 不允许再把“单个商品页面补采”当成库存主链；页面单采只用于补证据，不用于替代当天全量总表。
9. 库存待补信息必须与库存和 SN 同轮刷新，统一以 `inventory master` 当轮异常和入库缺口为准，不继续沿用旧待补快照。
10. 前端 `零售售价专区` 的库存字段必须直接映射同轮 `latest-standard-inventory-snapshot.json` 的 `currentStock / serialCount` 真值，不再走第二条库存计算线。
11. 销售价保前端口径固定写死：
   - `零售销售价保专区` 的 `申请日期` 必须取 `出库时间(outbound_date)`
   - `零售销售价保历史汇总` 只允许展示 `estimatedProtectionAmount > 0`
   - `estimatedProtectionAmount = 0` 的记录只能保留在后端历史库，不得进入前端历史汇总
12. `Lenovo Care / 智惠` 服务 SKU 规则固定写死：
   - 成本 `0` 视为正常
   - 不挂 `blocked_missing_cost`
   - 不进入硬件库存补成本任务
13. 质保结果固化规则固定写死：
   - 一旦 SN 已采到 `official_warranty_start / official_warranty_end`，后续任务不得回退成待补
   - `failed / captcha_required / 空结果` 不能覆盖已有成功质保时间
   - 后续定时任务只扫“前端当前不显示质保时间”的缺口和当天新入库 SN
   - 不再全量重采全部在库 SN

## 采集动作门禁

所有定时任务里，只要包含任何手工采集动作、电脑操控动作、可见页面动作，就必须遵守下面的统一门禁：

1. 先扫描目标当前状态
2. 判断当前是否处于可操作状态
3. 只执行一次手工单步动作
4. 动作后再次扫描状态
5. 状态正确才允许下一步

固定要扫描的内容：

- 当前窗口/标签页是不是目标
- 当前是不是已登录
- 当前是不是已加载完成
- 当前有没有白屏、转圈、验证码、403、安全验证、错误弹层、菜单栏收缩、非目标页
- 当前目标控件是不是稳定可见

明确禁止：

- 未扫状态就点
- 连点
- 双击补救
- 在错误状态下继续硬操作
- 用“还能点”代替“可以采”

统一阻塞口径：

- 只要扫描结果不是“目标页稳定可操作”，本轮任务必须立即停止
- 状态只能写：
  - `因缺原始数据阻塞`
  - `因页面/风控阻塞`
  - `已执行但未收口`
- 不能因为脚本已经启动，就假装采集还在继续推进


## 库存完整信息总表 SOP

### 目标

- 形成 `latest-inventory-master-snapshot.json`
- 口径固定为：
  - 一条记录 = 一个“当前在库”的 `SN`
  - 当前库存真值只认：
    - `商品库存统计`
    - `商品库存SN统计`
  - 追溯字段补证优先认：
    - `SN库存订单`
    - `库存流水`
    - `库存订单`

### 人工导出最小集合

必须导出：

1. `商品库存统计`
   - 必须勾选 `有库存商品`
2. `商品库存SN统计`
   - 必须与上一步尽量同时间、同筛选范围导出

按需补导：

3. `库存流水`
   - 用于补最近动作、历史业务时间、经手人、单据号
4. `SN库存订单`
   - 用于补 SN 级 `入库时间 / 单据号 / 交易人员 / 门店仓库 / 库位`
5. `库存订单`
   - 用于给没有 SN 粒度证据的单据做数量级兜底

### 本地合并执行顺序

1. 先重建标准库存快照
   - 基于 `商品库存统计 + 商品库存SN统计`
2. 再读取本地：
   - `latest-inventory-movements.json`
   - `latest-serial-overrides.json`
3. 再解析当轮人工导出：
   - `SN库存订单`
   - `库存订单`
4. 生成 `latest-inventory-master-snapshot.json`
5. 将结果同步到：
   - `apps/inventory-sync/artifacts`
   - `apps/web-cockpit/public/data`

### 字段补证优先级

1. `serialNumber / skuKey / productName`
   - 只来自 `商品库存SN统计`
2. `currentStock / sellableStock / occupiedStock / unsellableStock`
   - 只来自 `商品库存统计`
3. `inboundDate / inboundDocumentNumber / inboundOperatorName / locationName`
   - `SN库存订单` 优先
   - 其次 `serial overrides`
   - 再次 `库存流水`
   - 最后 `库存订单`
4. `latestMovementType / latestBusinessDate`
   - 优先本地已落盘的 `latest-inventory-movements.json`

### 完成判定

只有同时满足，才能说“完整库存信息总表已完成一轮同步”：

1. 有最新 `商品库存统计`
2. 有最新 `商品库存SN统计`
3. 主表已生成 `latest-inventory-master-snapshot.json`
4. 结果已同步到前端数据目录
5. 对缺口没有伪装完成，而是进入 `exceptions`

### 阻塞和边界

- 缺 `商品库存SN统计`：
  - 不能生成“每 SN 一行”主表，只能停在阻塞态
- 缺 `SN库存订单`：
  - 可以生成主表
  - 但必须把“缺单据号/交易人员/库位”的 SN 保留 warning
- 缺 `库存订单`：
  - 可以生成主表
  - 但无法对无 SN 的数量型库存订单做完整补证
- `库存流水` 不完整：
  - 允许只作为历史证据层
  - 不允许反推改写当前库存真值
- 已售/已调出且不在当前库存 SN 导出里的 SN：
  - 不写进主表 `rows`
  - 只写进 `exceptions`
- 人工 Excel 被二次编辑：
  - 视为不可信原始文件
  - 必须重新从智店通导出

   - 顺延到次日定时任务继续采集

### 完成判定

- 只有同时满足以下条件才算完成：
  1. 找到当天真实分销报价文件或当天有效截图
  2. 文件/截图已保存到本地
  3. 已完成解析入库
  4. 已生成对应快照
  5. 已记录未匹配商品和异常项
- 否则只能记为：
  - 部分完成
  - 等待文件
  - 待人工复核
  - 未完成

### 采集范围

- 只采：
  - 当天群内分销报价文件
  - 当天群内报价截图
  - 与门店库存商品相关的报价信息
  - 后续用于匹配 `SKU / 型号 / 配置` 的字段
- 只采有库存产品：
  - 无库存商品停止采集
  - 没有库存的报价不进入当日实时进货价更新
- 采集前必须先审计：
  - 今天本地是否已有同名文件
  - 文件日期是否变化
  - 群里是否有新增报价
  - 当前库存商品是否有新增 SKU
  - 当前是否有缺实时进货价商品
  - 是否有价格异常波动商品

- 不采：
  - 与电脑零售库存无关的闲聊
  - 无法确认日期的旧截图
  - 无法确认来源的转发零散文本
  - 没有报价字段、只有宣传话术的图片

1. 当天原始 Excel 文件
2. 当天原始截图
3. 当天原始图片中的报价表
4. 最后才是人工摘录文字
