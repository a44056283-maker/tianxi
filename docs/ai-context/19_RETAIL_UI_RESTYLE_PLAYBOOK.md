# 零售系统 UI 整改风格与实施流程

更新时间：2026-06-11
适用范围：`联想智慧零售驾驶舱 / 门店智能化管理系统` 全部后台页面

## 1. 整改目标

当前整改不是追求花哨视觉，而是统一成一套：

- 联想官网式的信息表达
- 京东/智店通式的业务台账
- 门店员工易看、易点、易核对的后台结构

落地目标：

1. 页面先看懂，再好看。
2. 一屏内优先完成对账、核销、出入库、价格复核。
3. 不再出现“深色大卡片堆叠、信息互相抢视线、按钮语义混乱”的情况。

## 2. 风格基线

### 2.1 视觉参照

- 联想官网：
  - 白底主内容区
  - 标题锚点明确
  - 版心稳定
  - 大块内容靠分区和留白区分，而不是靠重阴影
- 京东 / 智店通后台：
  - 强表格感
  - 强对齐
  - 列关系明确
  - 筛选、状态、金额、操作分区稳定

### 2.2 统一视觉规则

- 主内容区：白底或极浅灰底
- 边框：`#dbe5f1` 到 `#e7eef7`
- 正文主色：`#0f172a`
- 辅助文案：`#64748b`
- 激活蓝：`#2f6cde`
- 联想红锚点：`#e2231a`
- 阴影：只允许弱阴影，不允许厚阴影和悬浮卡片感

## 3. 布局骨架

### 3.1 页面层级

固定分成四层：

1. 页面标题区
2. 汇总区
3. 筛选/页签区
4. 主台账区

禁止把这四层混成一个大组件。

### 3.2 页面标题区

规则：

- 标题左侧允许使用联想红竖条做锚点
- 标题一行只放页面名
- 副说明只做一句业务边界解释
- 不在标题区塞操作按钮堆

### 3.3 汇总区

规则：

- 汇总卡必须平面化
- 每张卡只放：标签 / 数值 / 一句说明
- 金额与数量统一等宽数字
- 不做深色大卡，也不做复杂渐变

### 3.4 页签区

规则：

- 页签本身就是切换器，不再额外塞“切换查看”按钮
- 激活态只保留一套蓝色语义
- 页签内只放：
  - 类别名
  - 数量
  - 一句说明

## 4. 台账区规则

### 4.1 主表原则

- 主表是核心，不是卡片列表伪装
- 行和行之间靠细分隔，不靠厚边框
- 一条记录必须横向读完
- 禁止把一条记录拆成多个互相竞争的视觉盒子

### 4.2 列布局原则

优先使用固定列：

1. 时间 / 单号
2. 产品信息
3. 金额信息
4. 状态 / 来源 / 操作

如果是异常审计页：

1. 真实销售单
2. 真实产品信息
3. 当前代扫状态
4. 审计说明

### 4.3 产品信息区

规则：

- 多件套产品必须逐产品单列
- 类别、型号、MTM、SN 分行显示
- 不允许把多个产品压成一句长文本
- 客户、电话、代扫电话、数量等元信息放在产品区顶部的小型明细表中

### 4.4 金额信息区

规则：

- 金额右对齐或视觉右收
- 服务费、教育补、智享金分开展示
- 只突出主金额，其他金额做次级说明

### 4.5 状态区

规则：

- 状态用“文字 + 边框/底色”表达
- 操作按钮只保留 1 个主动作
- 门店、来源、SKU、匹配来源放入轻量键值区
- 告警文案只能做短句，不能写成长段说明

## 5. 禁止项

以下写法后续统一禁止：

- 深色大卡片堆叠成整页主视觉
- 一条业务记录被拆成多个风格不一致的块
- 页签里再套按钮
- 大量渐变、厚投影、发光边
- 把说明文字写得像段落而不是台账字段
- 用样式复杂度代替信息层级

## 6. 实施流程

每次整改按这个顺序执行：

1. 先定参照页
   - 联想官网信息页
   - 京东/智店通台账页
2. 拆当前页面
   - 哪些区块属于标题
   - 哪些区块属于汇总
   - 哪些区块属于筛选/页签
   - 哪些区块属于主表
3. 先改骨架
   - 不先调颜色
   - 先改层级、列结构、对齐关系
4. 再改视觉
   - 白底
   - 边框
   - 激活态
   - 状态色
5. 打开真实 `5174` 页面人工复核
6. 最后更新 ai-context 和记忆打包

## 7. 扩展到全系统时的顺序

建议按优先级同步：

1. `产品价保`
2. `库存详情`
3. `入库出库`
4. `报价来源`
5. `数据对接状态`

每个模块都按：

- 标题区
- 汇总区
- 页签区
- 主表区

逐层重做，不要一次性全站乱改。

## 9. 固定 UI 优化工作流

后续凡是说“优化 UI”“收口页面”“统一后台风格”，默认执行这套固定工作流：

1. `design-consultation`
   - 先定本轮页面边界、布局骨架、层级规则、筛选区/主表/详情区关系
2. `design-review`
   - 审当前页面的结构、间距、节奏、滚动区、首屏层级、视觉噪音
3. `design-critic`
   - 需要方案评审或觉得页面还不够像正式产品时再补
4. 落地修改
   - 先改骨架、表格、信息层级，再碰颜色和细节
5. `design-taste-frontend` / `taste-skill`
   - 只做最后一层去模板味、去 AI 味、细节收口
6. `qa`
   - 查错位、截断、滚动、密度失衡、响应式问题
7. `browser:control-in-app-browser`
   - 打开本地 `http://127.0.0.1:5174/` 做真实可见验收
8. `review`
   - 在准备合并或宣告收口前做代码层复核

最小可用组合：

- `design-consultation`
- `design-review`
- `qa`
- `browser:control-in-app-browser`

高质量收口追加：

- `design-critic`
- `design-taste-frontend`
- `review`

## 10. 每个 SKILL 的固定用途

- `design-consultation`
  - 用来定规则，不用来直接宣告页面已经好看
- `design-review`
  - 用来找问题和给出主修改方向，是后台页主技能
- `design-critic`
  - 用来挑“产品感不够”“专业感不够”的问题
- `design-taste-frontend`
  - 只做末端打磨，不单独主导后台页
- `design-shotgun`
  - 用在拿不准布局方向、需要 2 到 3 套方案比稿时
- `qa`
  - 用在改完后的真实验收
- `browser:control-in-app-browser`
  - 用在本地真实页面可见验证
- `review`
  - 用在最终代码审查

## 11. 固定简短触发词

后续如果用户只发下面这句，默认直接按本文件整套流程执行：

`按固定UI工作流收口这个页面`

等价理解：

- 这是后台页，不是营销页
- 先按台账/表格/筛选/详情区骨架处理
- 再做视觉收口
- 最后必须做真实本地页面验收
