# 14 Semi-Auto Automation Stack

更新时间：2026-05-13

## 结论

本项目不再尝试把智店通、微信、京东、联想官网这类无公开 API、强页面依赖、带风控的网站做成“纯终端后台自动采集”。

其中微信相关来源进一步收紧为：

- 分销群报价
- 灰渠公众号报价

都不再追求任何脚本自动采集，而是固定采用：

- 手动电脑操控
- 当天原始文件/原文落地
- 本地解析入库

正式采用：

- 浏览器可见半自动执行层
- 调度层
- 人工复核队列

目标不是 100% 无人值守，而是把每天重复劳动压到可控且真实的范围内。

## 适配结论

### 1. 采集核心

- 当前已登录 Chrome 可见会话
  - 用途：智店通、网页微信、京东、联想官旗、天猫等外部登录态页面的唯一采集入口
  - 必须复用老会话和已有标签页，禁止新浏览器、空白浏览器、新 Profile、清缓存、重登
- `Playwright` / `browser-use` / Browser / in-app browser / Puppeteer / Chromium launch
  - 只保留为本地前端验证、历史参考或非登录态页面研究工具
  - 不允许打开或推进智店通、网页微信、京东、联想官旗、天猫等外部采集页面

### 2. 调度层

- 继续使用现有：
  - `Codex automations`
  - `scripts/run_scheduled_task.sh`
  - `apps/inventory-sync/src/automation/scheduledTasks.ts`
- 不把调度层当采集层
- 调度层只负责：
  - 到点触发
  - 写报告
  - 产出半自动待办计划

### 3. 人工复核层

新增产物：

- `latest-semi-auto-execution-plan.json`

用途：

- 把“今天哪些任务必须人工/电脑操控继续做”结构化输出
- 让主线程和子代理都围绕同一份待办清单工作

## 为什么不走纯 CLI 后台采集

这类站点的共同问题：

1. 无稳定公开 API
2. 强依赖登录态或客户端
3. 页面结构经常变化
4. 有风控、403、验证码
5. 价格和活动规则需要人工判断

所以纯 CLI 爬虫会在你这个项目里反复失真：

- 任务“跑了”
- 但数据不真实
- 或页面一变就大面积失效

## 当前落地方式

### 已实现

1. 定时任务执行后自动生成：
   - 结构化任务报告
   - 半自动执行计划
2. 半自动计划当前覆盖：
   - 已锁定零售链接的实时价格复核
   - 缺失零售链接补链队列
   - 灰渠公众号当天原文缺口
   - 分销商群报价当天文件缺口
   - 智店通其它出库 SN 明细缺口
   - 保修待补队列

### 下一步

1. 让日报直接引用半自动执行计划
2. 让主线程按计划逐项核销
3. 微信相关来源继续坚持“手动电脑操控 + manual 文件 + 本地解析”路线，不再继续追求自动化采集

## 开源项目选型结论

以下项目适合作为参考，不直接替代本项目主链路：

1. `browser-use`
   - GitHub: <https://github.com/browser-use/browser-use>
   - 仅作为历史研究参考，不进入外部登录态采集主链路
2. `Playwright`
   - GitHub: <https://github.com/microsoft/playwright>
   - 仅用于本地前端验证或历史参考，不打开外部采集页面
3. `playwright-mcp`
   - GitHub: <https://github.com/microsoft/playwright-mcp>
   - 仅作为接口研究参考，不接管外部登录态页面
4. `n8n`
   - GitHub: <https://github.com/n8n-io/n8n>
   - 适合作为调度和流程编排参考

本项目当前决定：

- 先不引入新的外部云依赖
- 先在现有仓库中完成本地半自动执行层
