# 完整报价采集并上线定时任务

用途：交给 Claude 执行一轮真实可见界面报价采集。只有本轮真实完成后，才允许把流程设为后续定时采集任务。

## 执行总指令

默认使用中文。

按循环执行协议直接开始，不要询问是否开始、是否继续、是否执行下一步。收到本文件后，按顺序执行到任务完成、队列清空或出现真实阻塞。

你要完成的不是旧文件复核，也不是只写报告，而是：

```text
可见界面手动采集 -> 保存原始证据 -> 解析报价 -> 写入软件数据 -> 重建快照 -> 验证前端 -> 写回执 -> 成功后设定后续定时任务
```

## 绝对边界

这是“手动可见界面采集 + 软件数据对接”，不是脚本后台采集。

必须遵守：

- 先识别当前屏幕状态，再进行下一步。
- 只能在可见桌面界面里人工式操作微信、公众号或文件下载入口。
- 不得写脚本抓微信数据。
- 不得用 AppleScript、shell、Python、Node、坐标点击脚本或后台进程操作微信。
- 不得后台读取微信数据库、缓存、Cookie、登录态或容器私有数据。
- 不得自动点击 `Allow Once`、登录确认、验证码、安全验证或权限弹窗。
- 不得读取、输出、保存密码、Token、Cookie、验证码、密钥。
- 不得触发支付、删除、发货、退款、提交订单、账号权限变更。
- 不得把旧文件重跑说成今日实时采集。
- 不得只读旧 JSON 就说采集完成。

如果遇到登录、验证码、风控、`Allow Once`、权限弹窗、页面白屏、账号风险，停止该风险路径，写清阻塞状态，然后继续其它可做事项。

## 今日采集目标

日期以今天为准。

优先级：

1. 微信分销群当天报价 Excel。
2. 灰渠公众号当天报价原文或截图。

只采集有当天证据的报价。历史报价只能写成 `carry_forward_candidate`，不得写成今日新采。

## 第一步：当前界面状态识别

先识别当前桌面状态，写入临时判断：

```text
是否看到微信：
是否已登录：

是否看到今天报价入口：
是否存在权限、登录、验证码、Allow Once、风控、白屏：
下一步可执行动作：
```

如果看不到微信，先尝试打开已有微信应用窗口。若无法打开或未登录，记录 `blocked_page_or_login`。

## 第二步：可见界面手动采集

### 分销群报价

如果当前微信已登录：

1. 通过可见界面进入目标分销群。
2. 找今天的分销报价 Excel。
3. 通过可见界面保存或下载到人工输入目录。
4. 保存后的文件名建议：

```text
distributor-quotes-YYYY-MM-DD.xlsx
```

保存后必须确认：

```text
文件路径：
文件是否存在：
文件修改时间：
文件内容日期：
是否为今天：
```

### 灰渠公众号报价

如果公众号可打开：

1. 通过可见界面进入灰渠公众号。
2. 找今天报价原文、图片或可保存内容。
3. 保存到人工输入目录。
4. 保存后的文件名建议：

```text
gray-wholesale-YYYY-MM-DD.txt
gray-wholesale-YYYY-MM-DD.png
```

保存后必须确认内容日期，不得只看文件修改时间。

## 第三步：解析报价

只有原始证据已保存，才允许解析。

解析要求：

- 校验文件日期和内容日期是否为今天。
- 只对库存大于 0 的商品参与匹配。
- 标题、型号、配置不确定时不得强写。
- 不确定项进入人工复核队列。
- 解析结果必须保留原始证据路径。

如果项目已有解析命令，优先使用项目现有命令。不得临时发明一条不可追溯的数据链路。

## 第四步：写入软件数据

解析成功后，按项目既有数据链路写入。

允许更新：

```text
apps/web-cockpit/public/data/latest-distributor-quotes.json
apps/web-cockpit/public/data/latest-gray-wholesale-quotes.json
apps/inventory-sync/artifacts/
docs/claude-scheduled-tasks/receipts/
```

不得在没有原始证据的情况下写正式快照。

不得写入伪造价格。

## 第八步：成功后设定定时任务

只有同时满足以下条件，才允许设定后续定时任务：

```text
本轮 status = real_completed
至少一个当天原始证据文件已保存
解析成功
软件数据已写入
快照重建成功
前端验证完成或已说明无法验证的原因
未出现登录、验证码、Allow Once、权限确认、风控阻塞
```

如果满足条件，创建或登记后续定时采集任务：

任务名：

```text
daily-visible-quote-collection
```

建议执行时间：

```text
11:30 分销群报价首轮
13:45 分销群报价补查
11:50 灰渠公众号首轮
13:50 灰渠公众号补查
```

定时任务必须执行同一条闭环：

```text
可见界面状态识别 -> 手动可见界面采集 -> 保存原始证据 -> 解析 -> 写入 -> 重建 -> 前端验证 -> 回执
```

定时任务失败也必须写回执，不允许静默失败。

如果 Claude 当前环境不能真正创建系统级定时任务，则写出：

```text
scheduledTaskCreated: false
blockingReason: 当前 Claude 环境不能创建可靠系统级定时任务
handoffPath: docs/claude-scheduled-tasks/receipts/YYYY-MM-DD-定时任务创建交接.md
```

并生成交接文件，说明需要用户或 Codex 用哪个调度器创建。

## 第九步：没有成功时不得上线

如果本轮只是：

- 读了旧 JSON。
- 只检查了已有文件。
- 没看到微信界面。
- 没保存当天原始证据。
- 没完成数据写入。
- 没完成快照重建。
- 没完成前端验证。
- 遇到 Allow Once、登录、验证码、风控。

则不得说“已设定以后定时采集”，只能写：

```text
scheduledTaskCreated: false
status: executed_not_closed 或 blocked_*
```

并写清下一步从哪里继续。
