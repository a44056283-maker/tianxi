## 2. 先读顺序

软件开发任务先补充读取本机已安装的方法论技能：

1. `gstack`
2. `gstack-codex`
3. `superpowers`
4. `superpowers-writing-plans`
5. `superpowers-test-driven-development`
6. `superpowers-systematic-debugging`
7. `superpowers-verification-before-completion`
8. `superpowers-subagent-driven-development`
9. `taste-skill`

要求：

- 只要任务属于“做软件 / 改代码 / 设计实现 / 调试修复 / 发版验收”，先读以上技能，再进入项目上下文。
- 这是永久软件开发要求，不是建议：所有非极小软件任务必须先按 `superpowers/gstack` 走计划、实现、验证和真实页面 QA。
- 禁止把代码先改完再事后补流程当成默认工作方式；如果发生，必须立即停止扩散，补写计划并回到 `superpowers/gstack` 流程。
- `gstack` 侧重产品化工作流、浏览/验收/发布节奏；`superpowers` 侧重计划、TDD、系统化调试和完成前验证。
- 只要任务涉及前端界面、交互、布局、视觉统一、页面重构或最终 UI 收口，必须补读 `taste-skill`，并把它作为最后的 UI 审核优化技能。
- 只要任务明确属于“后台页面 UI 优化 / 页面重构 / 视觉收口 / 交互整理”，固定 UI 工作流写死为：
  1. 先读 `docs/ai-context/19_RETAIL_UI_RESTYLE_PLAYBOOK.md`
  2. 先用 `design-consultation` 定本轮页面规则
  3. 再用 `design-review` 找结构、层级、间距、节奏问题
  4. 需要评审时补 `design-critic`
  5. 页面主修改完成后，必须用 `taste-skill` 做最后一轮去模板味/去 AI 味收口
  6. 最后用本地前端真实页面 QA 收尾，不得只看代码就宣告 UI 已完成
- 这条 UI 工作流属于永久规则；后续进入 UI 任务时，默认直接执行，不再每轮重新发明流程。
- 用户如果只说“收口 UI 界面”或给出项目约定的 UI 短提示词，默认等价于触发以上整套 UI 工作流。
- 如果这些技能与本项目真实业务边界冲突，以本仓库 `AGENTS.md` 和 `docs/ai-context/*` 的真实业务边界为准。

固定默认流程：

1. 非极小改动，先读并优先执行 `superpowers-writing-plans`，把目标、范围、文件、验证写清楚后再改代码。
2. 进入实现时，优先遵循 `superpowers-test-driven-development`：
   - 先写或先明确失败验证
   - 再做最小实现
   - 再跑通过验证
3. 遇到异常、回归、原因不清时，优先切到 `superpowers-systematic-debugging`，禁止靠猜测连续打补丁。
4. 准备宣告“完成”前，必须按 `superpowers-verification-before-completion` 做最终核验，尤其要核：
   - 真实数据链是否仍然成立
   - 移动端/前端是否真机或真实浏览器可用
   - 定时任务/采集任务是否只是脚本通过而非业务完成
5. 只要本轮涉及前端/UI，最终核验后还必须执行一次 `taste-skill` 视角的 UI 审核优化，再决定是否收口：
   - 检查是否仍有模板味、重复布局、层级混乱、视觉密度失衡、风格不统一
   - 优先修正页面第一屏、导航结构、卡片/表格/表单的一致性
   - 不得跳过这一步直接把“功能做完”写成 UI 已收口
6. 中高风险实现、共享模块修改、重要 UI/交互改动，结束前追加一次 `gstack-codex` 第二意见审查。
7. 需要浏览验收、发布节奏、回归检查、上线收口时，再补读 `gstack` 对应技能；不要把它替代本项目的真实业务边界。

## 3. 工作边界

- 只在项目目录内创建/修改代码和文档。
- 不操作真实账号的支付、删除、提交、权限变更。
- 不读取或保存密码、Token、Cookie、验证码。
- 第三方网站可做可见内容审计和人工辅助采集，但不应成为生产主链路。
- 所有依赖登录态的网页采集必须使用用户当前已经登录的默认 Chrome 会话；禁止重新打开新的浏览器、打开空白浏览器、新建 Chrome Profile、清理登录缓存、主动退出账号或用无登录态窗口替代已有会话。
- 外部采集页面禁止使用 Browser/in-app browser/browser-use/Playwright/Puppeteer/Chromium launch 打开；这些工具只能用于本地前端视觉验证，不能用于智店通、网页微信、京东、联想官网、天猫、淘宝、联想保修页采集。
- 遇到验证码、403、滑块、安全验证，立即停止该路径并记录。
- 允许的登录恢复边界只限于“恢复到已有账号的正常工作会话”：
  - 智店通转圈/白页时，可关闭异常目标页，回到首页重新点登录。
  - 智店通被动退出、跳回登录页或会话失效时，不得直接判阻塞；必须先按固定登录恢复顺序尝试一次。
  - 如果登录页要求输入手机号，只允许输入固定手机号 `15637798222` 进入下一步。
  - 智店通标准恢复顺序写死为：`打开登录页 -> 输入手机号 15637798222 -> 下一步 -> 点一次密码输入区域 -> 选择浏览器已保存的密码候选 -> 确认登录`。
  - 如果系统弹出浏览器已保存的密码候选，只允许点现有候选完成登录，不手写、不保存新密码。
  - 如果出现短信验证码、二次认证、滑块、人机验证，立即停止并通过飞书群提醒用户手动处理。
  - 京东、联想官网、天猫、淘宝、网页微信及其他采集站点一旦出现登录失效、验证码、403 或安全验证，也必须走飞书提醒，不得硬闯。

## 3.1 全采集统一动作门禁

- 本项目所有采集，不分微信、公众号、智店通、京东、联想官网、天猫、淘宝、桌面程序，统一先走同一条动作门禁：
  1. 先扫描目标当前状态
  2. 判断当前是否处于可操作状态
  3. 只执行一次手工单步动作
  4. 动作后再次扫描状态
  5. 只有状态正确，才允许下一步
- 这里的“扫描状态”必须至少确认：
  - 当前窗口/标签页是不是目标对象
  - 当前是不是已登录、已展开、已完成加载
  - 当前有没有白屏、转圈、验证码、403、安全验证、底部菜单栏收缩、错误弹层
  - 当前目标控件是不是稳定可见、不是滚动中或重绘中
- 明确禁止：
  - 看不清状态就直接点
  - 连点
  - 双击补救
  - 盲点正文区、文章区、表格区、底部菜单区
  - 在白屏、转圈、错误页、非目标页、登录页状态下硬操作
- 统一阻塞口径：
  - 只要状态扫描结果不是“目标页稳定可操作”，本轮采集就必须立即停下
  - 任务状态只能写阻塞，不能假装继续推进
- 智店通专项恢复例外：
  - 如果明确是“目标页持续转圈/白页”而不是正常登录页，可先关闭异常页，重新打开首页并按上面的登录恢复边界重进一次。
  - 重进后仍转圈、仍需短信验证或仍无法进入业务页，立即写 `blocked_page_risk` 并触发飞书提醒。
- 这条规则是项目级硬规则，不是微信专项例外。

## 价格采集顺序写死规则

- 这条规则适用于手工采集、半自动采集、定时任务采集，默认强制执行，不需要用户每次重复提醒。
- 如果当前页面或上一轮操作已经误入京东全站搜索、错误年份搜索或非预设店铺路径，必须废弃该路径候选，回到预设自营店 / 用户指定店铺或联想商城首页重新执行。
- 京东来源固定顺序：
  1. 先采用户指定店铺
  2. 指定店铺确实没有，再采京东全站
- 新入库 / 新上市电脑 SKU 的京东补链必须先进入预设自营店或用户指定店铺做店内检索和分类筛选，不允许直接从京东全站搜索开始。
- 联想来源固定顺序：
  1. 先采联想官旗
  2. 官旗确实没有或已下架，再用天猫 / 淘宝替代
- 新入库 2026 新品搜索词严禁主动加入 `2025`。除非商品标题、用户给定链接或页面证据明确写有 `2025`，否则只用系列型号和核心配置检索。
- 搜索顺序固定：
  1. `型号`
  2. `型号 + 核心配置`
  3. `型号 + 核心配置 + 颜色`
- 首轮禁止直接把整串完整配置硬搜到底；必须先用型号打开系列结果，再在详情页或规格区核对子配置。
- 京东/联想系列商品必须继续点到正确的 `处理器 / 颜色 / 版本 / 存储` 子配置；系列对但子配置错，视为错链错价，不得写回。
- 京东详情页如果只显示国补后/到手价，且页面叠加了首发满减、学生券、教育补或 `PLUS` 立减：
  1. 当前轮主零售价先与匹配官旗零售价同步
  2. 页面可见到手价、国补金额、`PLUS` 立减、首发优惠只做拆分备注
  3. 待用户补充营销活动材料后，再同步到零售价英雄卡
- 只有没有匹配官旗零售价时，才允许按京东页面明确可见的国补金额与立减金额反推临时 JD 主价；未在详情页明示的教育补/海报活动不允许硬猜。
- 配件门店零售价固定规则：
  1. 先取当天真实采集价
  2. 再按 `采集价 × 1.1`
  3. 最终价格统一取 `9` 结尾
  4. 不再使用旧的配件阶梯加价口径
- 电脑 / 手机 / 平板门店价固定规则：
  1. 京东自营、联想官旗采集零售价必须取平台券后价或秒杀后价；教育补贴只做备注，不覆盖零售价。
  2. 电脑门店基准价优先取 `京东自营价 + 联想官旗价` 的平均值；缺一方时用已有有效平台价，官旗缺失可用天猫/淘宝补充价兜底。
  3. 电脑门店售价 = 平台平均零售价 + 电脑加价逻辑，再统一收尾为 `99`。
  4. 手机 / 平板门店售价 = 两个平台有效零售价平均值，再统一收尾为 `99`，不叠加电脑加价。
  5. 所有电脑、手机、平板门店零售价必须以 `99` 结尾，不允许出现其它尾数。
  6. 任意电脑、手机、平板 SKU 入库后触发零售价更新时，必须同步按以上规则完善门店售价并刷新前端；不得只更新平台采集价而留下门店价旧值或空白。
- 防流失补贴价规则：
